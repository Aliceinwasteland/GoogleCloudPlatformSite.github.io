Mobile Lösungen auf der Google Cloud Platform

Einführung

Angesichts der Tatsache, dass eine Milliarde Menschen Smartphones und Tablets in ihrem Alltag nutzen, besteht ein riesiger Markt und ein wachsender Bedarf für mobile Lösungen. Die Nutzer können aus Hunderttausenden von Anwendungen wählen, deshalb müssen erfolgreiche mobile Anwendungen überzeugende, interaktive und gemeinsame Nutzererfahrungen bieten. Dies erfordert in der Regel, dass Back-End-Komponenten oder -Dienste die Anwendung mit relevanten Daten versorgen, Push-Benachrichtigungen bereitstellen, Interaktionen zwischen Nutzern ermöglichen und mehr.

Mithilfe der Google Cloud Platform können Sie auf einfache Weise ein mobiles Back-End für Ihre mobile Lösung erstellen. Entweder fangen Sie klein an und hoffen darauf, eine große Anzahl von Nutzern zu gewinnen, oder Sie verfügen bereits über einen großen Kundenstamm, den Sie jetzt über eine neue mobile Erfahrung bedienen möchten. In jedem Fall können Sie ein Back-End entwickeln, das sich für den wachsenden Bedarf skalieren lässt. Wenn Sie die Google-Infrastruktur nutzen, können Sie sich auf die Implementierung Ihrer Anwendungsszenarien konzentrieren und müssen sich nicht mit der Verwaltung von Geräten oder dem Lastenausgleich befassen.

Anforderungen

Typische Anforderungen für eine überzeugende mobile Lösung sind:

  • Unterstützung für Android- und iOS-Geräte über native Anwendungen
  • Speicherung, Abruf und Verarbeitung von Daten außerhalb von Mobilgeräten
  • Orchestrierung von Push-Benachrichtigungen auf Android- und iOS-Geräte
  • Geografische Ortung und Umgebungssuche
  • Nutzerauthentifizierung
  • Hohe Skalierbarkeit

Diese Anforderungen erfüllen Sie problemlos, wenn Sie Ihre mobile Lösung auf der Google Cloud Platform erstellen.

Umfang

In diesem Artikel werden die wesentlichen Komponenten einer mobilen Lösung vorgestellt, die auf der Google Cloud Platform entwickelt wird. Dazu gehören sowohl Komponenten, die die meisten mobilen Lösungen gemeinsam haben, als auch optionale Komponenten, die in spezifischen Szenarien nützlich sein können.

Der vorliegende Artikel ist für Softwarearchitekten und -entwickler bestimmt, die es sich zum Ziel gesetzt haben, ihr eigenes mobiles Back-End zu erstellen und die Kommunikation zwischen mobilen Clients und dem Back-End zu konzipieren. Die Erstellung der eigentlichen mobilen Clients wird in diesem Dokument nicht behandelt, mit Ausnahme von Informationen bezüglich der Kommunikation mit dem Back-End, der Authentifizierung von Nutzern und des Empfangs von Push-Benachrichtigungen.

Überblick

Die vorgeschlagene mobile Lösungsarchitektur umfasst folgende Hauptkomponenten:

  • Ihre mobilen Clients – eine beliebige Kombination aus Android-, iOS- und HTML/JavaScript-Clients
  • Ihre mobile Back-End-Anwendung, die für die Übermittlung von Anfragen von den Clients verantwortlich ist
  • Die Kommunikationsebene zwischen den mobilen Clients und dem Back-End
  • Verschiedene Cloud-Dienste für die Speicherung der Anwendungsdaten, für die Übermittlung von Push-Benachrichtigungen usw.

Dieses Dokument führt Sie durch die Arbeitsschritte, die für die Zusammenstellung einer mobilen Lösung erforderlich sind. Am Ende finden Sie schließlich ein vollständiges Architekturdiagramm.

Ihr mobiles Back-End[1] in der Cloud

Wenn Sie über ein eigenes Back-End verfügen, können Sie benutzerdefinierten Code auf dem Back-End ausführen und sind nicht darauf beschränkt, lediglich Drittanbieterdienste aus Ihren mobilen Clientanwendungen aufzurufen.

Mithilfe der Google Cloud Platform können Sie problemlos Ihr eigenes Back-End in der Cloud realisieren. Google App Engine ist eine ideale Plattform für die Ausführung Ihres Codes, insbesondere von Code, der von Ihren mobilen Clientgeräten aufgerufen werden kann.

Primäre Funktion Ihres benutzerdefinierten Codes im Back-End

Der in diesem Dokument vorgestellte Ansatz zur Erstellung des Back-Ends ermöglicht es Ihnen, sich auf die Entwicklung Ihrer Kernszenarien zu konzentrieren. Ihre beiden Hauptaufgaben bestehen in der Definition der domainspezifischen Ressourcen und in der Implementierung der benutzerdefinierten Logik, die für spezielle Vorgänge auf diesen Ressourcen benötigt wird. Ihre mobilen Clients werden problemlos auf diese Ressourcen zugreifen und diese Vorgänge über Ihre API aufrufen können. Die API wiederum kann, wie weiter unten beschrieben, einfach aus dem mobilen Back-End freigeschaltet werden.

Ihr mobiles Back-End ist in der Lage, bestimmte Ressourcen von externen Diensten aufzurufen, ohne diese innerhalb der Anwendung zu speichern. In der Regel wird Ihre mobile Lösung jedoch mindestens über einige Ressourcen verfügen, die als Teil der Lösung verwaltet und persistiert werden.

Wenn Sie zum Beispiel eine Anwendung erstellen, die Bewertungen enthält, können Sie "Bewertung" als eine Ihrer "Ressourcen" definieren. Wenn Ihre Anwendung mobiles Einkaufen ermöglicht, können Sie im Inventar Bewertungen von Produkten aufnehmen. Wenn Ihre Anwendung eine Liste mit Filmen enthält, können Sie Filmkritiken aufnehmen. Wenn Sie ein Gelegenheitsspiel entwickeln, können Sie CompletedPlay als Entität mit Attributen wie Punktzahl, Datum, Nutzer-ID usw. definieren.

Anschließend können Sie Vorgänge auf diesen Ressourcen festlegen, zum Beispiel das Einfügen einer neuen Bewertung, das Hinzufügen der Nützlichkeitseinstufung einer Bewertung, das Abrufen von Bewertungen sortiert nach ihrer Nützlichkeitseinstufung oder das Abrufen der Highscores des Tages.

Je nach Szenario und Anforderungen wird die Implementierung dieser Vorgänge und des benutzerdefinierten Codes nicht immer leicht sein. Neben der synchronen Verarbeitung einfacher API-Anfragen von den Clients können Sie mit Ihrem Code noch eine Menge anderer Dinge realisieren. Sie können beispielsweise Prozesse im Hintergrund, Suchvorgänge in der näheren geografischen Umgebung oder Push-Benachrichtigungen implementieren. Alle diese Optionen stehen Ihnen offen, wenn Sie Ihr mobiles Back-End auf der Google Cloud Platform ausführen. Als Resultat können Sie sich auf das Schreiben von anwendungsspezifischem Code anstelle von Infrastrukturcode konzentrieren.

Dinge, mit denen Sie sich kaum beschäftigen müssen

Wenn Sie Ihren mobilen Back-End-Code nach den empfohlenen Vorgaben in Google App Engine ausführen, profitieren Sie von den folgenden Vorteilen der Google-Infrastruktur:

  • Automatische Skalierbarkeit
  • Automatischer Lastenausgleich über mehrere Instanzen des mobilen Back-Ends
  • Statische Bereitstellung von Inhalten und statisches Caching
  • Zuverlässigkeit, Leistungsstärke und Sicherheitsfunktionen
  • Überwachung und Verwaltung
  • Konfigurierbarer DoS-Schutz
  • Integration mit Google-Konten für die OAuth2-Authentifizierung (sowohl für Android- als auch iOS-Clients)

Kommunikation zwischen den mobilen Clients und dem mobilen Back-End

Es gibt viele mögliche Konzepte für die Kommunikation zwischen mobilen Clients und dem Back-End. Im Laufe des letzten Jahrzehnts hat sich das REST-basierte Modell jedoch immer mehr durchgesetzt.

Die Google Cloud Platform bietet eine leistungsfähige neue Technologie namens Google Cloud Endpoints[2], die nicht nur die Freischaltung der REST API aus dem mobilen Back-End, sondern auch die Nutzung der REST APIs aus Android-, iOS- und JavaScript-Clients vereinfacht. Dieses Vorgehen unterstützt auch die OAuth2-basierte Authentifizierung, damit die Identität des Nutzers im mobilen Back-End-Code hinterlegt ist. Der Cloud Endpoints-Dienst nutzt die Google-Infrastruktur, deshalb können Sie sich darauf verlassen, dass eine bedarfsorientierte Skalierung erfolgt. Sie können die bereitgestellten Tools für die automatische Generierung des benötigten Standardcodes verwenden, damit Sie sich auf das Schreiben von spezifischem Code für Ihre Anwendungsszenarien konzentrieren können.

In der Google Cloud Endpoints-Dokumentation finden Sie Informationen zur Annotierung Ihres Anwendungscodes, damit Ihre API über Google Cloud Endpoints freigeschaltet werden kann, des Weiteren Anleitungen zur Konfiguration der OAuth2-Authentifizierung und zur Erstellung stark typisierter Android- und iOS-Clients sowie wenig komplexer JavaScript-Clients. Wenn Sie Eclipse verwenden, profitieren Sie von zusätzlichen integrierten Hilfstools, die über das Google-Plug-in für Eclipse bereitgestellt werden.

Wenn Sie Ihr mobiles Back-End ohne Google Cloud Endpoints erstellen, treffen zwar viele Abschnitte dieses Dokuments zu. Allerdings gehen wir bei der vorgeschlagenen Lösung davon aus, dass Google Cloud Endpoints verwendet wird.

Mobile Lösung zusammenstellen

Grundlagen

Wie bereits oben erwähnt, umfasst die vorgeschlagene mobile Lösungsarchitektur folgende Hauptkomponenten (Abb. 1):

  1. Mobile Android- und/oder iOS-Clients
  2. Google Cloud Endpoints für die Kommunikation zwischen den Clients und dem Back-End über die REST API und mit optionaler OAuth2-Authentifizierung
  3. Ihr mobiler Back-End-Anwendungscode, der in Google App Engine ausgeführt wird und für die Übermittlung von Anfragen der Clients zuständig ist

Daten speichern

Eine typische Anforderung für eine mobile Lösung mit Back-End besteht darin, dass die Daten außerhalb von Clientgeräten gespeichert werden[3]. Diese Daten können in zwei Gruppen (Abb. 2) unterteilt werden:

  1. Große und typischerweise binäre Objekte wie Bilder

    Diese Objekte können Teil Ihrer Lösung sein, zum Beispiel Produktbilder für ein dynamisches Inventar von Geschenken, die über Ihre mobile Anwendung angeboten werden. Möglicherweise möchten Sie Ihren Nutzern auch das Hochladen von eigenen Bildern, Avataren oder Grafiken gestatten. In beiden Fällen können Sie für das Speichern dieser Art von Daten Google Cloud Storage nutzen, einen Dienst, der Objekte und Daten bis zu Terabytegröße speichert. Siehe auch Bilder bereitstellen und verarbeiten weiter unten.

  2. Detailgenaue Eigenschaften und Entitäten

    Zu diesen Eigenschaften und Entitäten zählen zum Beispiel Informationen über das letzte abgeschlossene Level in einem mobilen Spiel, Informationen über Nutzer, die Registrierung von Nutzergeräten für Push-Benachrichtigungen, Aufzeichnungen von In-App-Käufen und andere ähnliche anwendungsspezifische Daten.

    Die Eigenschaften dieser Entitäten können auch eine Referenz auf Objekte, die in Google Cloud Storage gespeichert sind, umfassen, zum Beispiel einen Objektnamen und optional den Bucket-Namen oder die URL.

    Wenn Ihr mobiles Back-End in Google App Engine gehostet wird, würde sich als Speicherplatz für diese Art von Daten der App Engine-Datenspeicher anbieten. Er bietet einen schemalosen NoSQL-Objektdatenspeicher[4] mit einer Abfrage-Engine und atomischen Transaktionen. Diese Entitäten werden häufig Ressourcen zugeordnet, die über die Google Cloud Endpoints API freigeschaltet werden. Bei Verwendung geeigneter Annotationen können Sie mit dem Google-Plug-in für Eclipse ein Codegerüst für typische list/get/insert/update/remove-Operationen mithilfe des App Engine-Datenspeichers erzeugen.

Datenzugriff mit Memcache optimieren

Wenn Ihre mobilen Clients häufige Anfragen an das Back-End zum Abruf derselben Daten stellen, wäre es nicht optimal, diese Daten bei jeder Anfrage aus dem persistenten Speicher abzurufen. Wenn Sie zum Beispiel eine mobile Anwendung mit Filmkritiken erstellen, ist es wahrscheinlich, dass die Nutzer Kritiken über eine Handvoll Filme abrufen, die derzeit in den Kinos laufen, und wesentlich seltener Kritiken für die übrigen Tausenden von Filmen, die das Back-End im persistenten Speicher vorhält.

Ein Back-End, das in Google App Engine ausgeführt wird, kann einen verteilten speicherinternen Datencache-Dienst namens Memcache nutzen, um Daten, auf die häufig zugegriffen wird, zu speichern und abzurufen (Abb. 3). Mit Memcache können Sie Ihre mobile Lösung nicht nur für höhere Leistung und bessere Skalierung konfigurieren, sondern auch die Kosten für den Zugriff auf den App Engine-Datenspeicher senken, da Ihre Anwendung weniger Anfragen an den Datenspeicher senden wird

Asynchrone Verarbeitung mit Aufgabenwarteschlangen

Gelegentlich ist es nicht möglich oder wünschenswert, alle Arbeiten im Zusammenhang mit einer Anfrage, die von einem mobilen Client gesendet wird, durchzuführen, bevor die Antwort an das Gerät des Nutzers zurückgesendet werden muss. Gehen wir zum Beispiel davon aus, dass Sie eine soziale Anwendung entwickeln, die die Freunde eines Nutzers benachrichtigt, wenn der Nutzer in einem Restaurant eincheckt. Es empfiehlt sich, das Check-in-Ereignis synchron zu erfassen, zusammen mit der Tatsache, dass die Benachrichtigung verarbeitet werden muss, und anschließend die Bestätigungsantwort sofort an den Client zu senden, ohne die Verarbeitung aller Benachrichtigungen abzuwarten.

Ihr Code, der in Google App Engine ausgeführt wird, kann einen Dienst namens Task Queues (Aufgabenwarteschlange) nutzen, um Informationen über die asynchron durchzuführenden Aufgaben in die Warteschlange zu stellen. Im obigen Beispiel könnte Ihre Anwendung eine Aufgabe zur Benachrichtigung von Max' Freunden in die Warteschlange aufnehmen, dass Max um 7:35 Uhr im Restaurant Trendy Perks eingecheckt hat.

Normalerweise würden Sie den Dienst "Task Queue" so konfigurieren, dass die Aufgaben in der Warteschlange (Push-Modell) als HTTP-Anfragen für eine ausgewählte URL an den Code gesendet werden, der in Google App Engine ausgeführt wird. Ihre Anwendung kann diese Anfragen dann mithilfe von normalen Webanfrage-Handlern verarbeiten. Im obigen Beispiel könnte der Anfragen-Handler die Aufgabendaten aus der HTTP-Anfrage abrufen, nach Max' Freunden suchen und dann die einzelnen Freunde benachrichtigen, indem eine Push-Benachrichtigung gesendet wird. Sie können vorgeben, dass die Aufgaben durch dieselben Google App Engine-Instanzen verarbeitet werden, die Anfragen von mobilen Clients verarbeiten, oder durch dedizierte Instanzen.

Alternativ können Sie mit Ihrem Code die Aufgaben aus den Warteschlangen abrufen. Dies kann entweder mithilfe des mobilen Back-End-Codes erfolgen, der in Google App Engine ausgeführt wird, oder mit dem Code außerhalb von App Engine, der zur Verarbeitung von Aufgaben durch Google Compute Engine-Instanzen verwendet wird.

In jedem Fall ist die Verarbeitung von Aufgaben der zweithäufigste Einsatzzweck von in Google App Engine ausgeführtem benutzerdefinierten Code neben der Verarbeitung von Anfragen aus mobilen Clients über Google Cloud Endpoints, die weiter oben erwähnt wurde.

Push-Benachrichtigungen

Push-Benachrichtigungen sind für die gesamte Nutzererfahrung auf Mobilgeräten sehr wichtig. Abhängig von der Clientplattform kann Ihre mobile Lösung diese Technologie zur Anzeige von Pop-ups und anderen Benachrichtigungen auf den Geräten der Nutzer verwenden, auch wenn die mobile Anwendung nicht ausgeführt wurde. Sie können damit die mobile Clientanwendung auch weiter mit relevanten Daten versorgen.

Wenn der mobile Back-End-Code in Google App Engine ausgeführt wird, können Sie das Senden von Push-Benachrichtigungen an Ihre Nutzer über die Verwendung der folgenden Technologien orchestrieren:

  • Google Cloud Messaging für Android – ein Google-Dienst, mit dem Sie Daten an Nutzer senden können, auf deren Android-Geräten Ihre Clientanwendung installiert ist. Bei Verwendung von Eclipse können Sie die Rohfassung des Codes vom Google-Plug-in für Eclipse erzeugen lassen.
  • Sockets API[5] – zum Senden von Push-Benachrichtigungen mit dem Apple®-Push-Benachrichtigungsdienst an iOS-Nutzer

Ein Beispiel dafür wäre das Senden von Push-Benachrichtigungen an die Nutzer Ihrer mobilen Lösung, wenn deren Freund in ein Restaurant eincheckt oder eine Bewertung postet (Abb. 5).

Statische Inhalte bereitstellen

In manchen Fällen können Sie den Vorteil nutzen, statische Inhalte in das Google App Engine-Anwendungspaket aufzunehmen und dieses dann mithilfe der Google App Engine-Infrastruktur unter Verwendung von dedizierten Servern und Cachespeichern ohne jegliche Beteiligung Ihres mobilen Back-End-Codes bereitstellen zu lassen. Diese Vorgehensweise betrifft in der Regel eher HTML/JavaScript-Clients als native iOS- oder Android-Anwendungen und ist auf solche Ressourcen wie Bilder, CSS-Stylesheets oder JavaScript-Code im Browser anwendbar.

Bilder bereitstellen und verarbeiten

Wenn Ihre mobile Lösung eine verwaltbare Anzahl von statischen Bildern beinhaltet, können Sie diese (häufig in mehreren Auflösungen) in ein Paket mit Ihren nativen Android- und iOS-Anwendungen aufnehmen. Dies gilt zum Beispiel für verschiedene Symbole, die zu Ihrer Anwendung gehören und zur Erstellung der Benutzeroberfläche Ihrer Clientanwendungen herangezogen werden.

Eine andere Kategorie von Bildern sind Bilder, die nicht Teil Ihrer Anwendungen sind, sondern sich eher auf die Daten beziehen, die von Ihrer Anwendung verarbeitet werden, zum Beispiel Miniaturansichten von Filmen in Ihrer filmbezogenen Anwendung, Bilder von Produkten, die über Ihre Anwendung als Geschenk verteilt werden, oder Avatare in Ihrem Spiel bzw. Ihren sozialen Anwendungen. Sie können solche Bilder in für Ihre mobilen Clients optimierten Auflösungen und Formaten speichern, sie in Google Cloud Storage hochladen und anschließend den mobilen Clients direkte URLs zu diesen Bildern liefern. Dieser Ansatz ermöglicht eine ausgezeichnete Skalierung, weil die mobilen Clients die Bilder direkt ohne Beteiligung des mobilen Back-End-Codes aus dem skalierbaren Cloudspeicher abrufen können.

In manchen Fällen ist zusätzlich zur Speicherung eine Be- und Verarbeitung von Bildern wünschenswert. Vielleicht möchten Sie die Bilder je nach Auflösung der Clientgeräte in dynamisch angepassten Größen bereitstellen. In solchen Fällen können Sie den Google App Engine-Bilderdienst für Größenänderung, Drehung, Spiegelung und Zuschneiden von Bildern sowie zur Verbesserung von Fotos anhand eines vordefinierten Algorithmus verwenden. Diese Operationen können mit Bildern aus Google Cloud Storage und anderen Quellen durchgeführt werden. Die modifizierten Bilder können dann aus der dynamischen, von Google bereitgestellten Infrastruktur für die Übermittlung von Bildern direkt, ohne Beteiligung Ihres mobilen Back-End-Codes, an die Clients geliefert werden. Ihr Code hat nur die Aufgabe, die Images Service API für den Abruf der passenden Bild-URL einzusetzen, und diese URL an die mobilen Clients zu übermitteln.

Volltextsuche und geografische Umgebungssuche (standortbasiert)

Wenn Sie den Nutzern die Volltextsuche in von Ihrer mobilen Lösung verwalteten strukturierten Daten ermöglichen wollen, können Sie die Search API[6] von Google App Engine nutzen. Wenn Ihre Anwendung beispielsweise den Zugriff auf ein großes und dynamisches Inventar von Objekten vorsieht, profitieren die Nutzer möglicherweise davon, dass sie Objekte durch die teilweise Eingabe des Namens oder der Beschreibung eines Objekts finden können, auch wenn Sie eine komfortable Touch-Oberfläche mit Kategorisierung, Seitenverwaltung usw. bereitstellen.

Für die Verwendung der Search API können Sie "Dokumente" mit suchbaren Daten erstellen und anschließend diese Dokumente in einen Index aufnehmen. Ihr mobiles Back-End kann dann Suchabfragen, einschließlich Volltextsuchabfragen, erstellen und diese auf die "Dokumente" im Index anwenden. Im obigen Beispiel könnte jedes Objekt im Inventar einem "Dokument" im Index mit Eigenschaften wie Name, Beschreibung usw. zugeordnet werden.

Ihre mobile Lösung kann außerdem eine geografische Umgebungssuche durchführen, wenn Sie zum Beispiel eine Liste mit Geschäften in der Nähe abrufen möchten. Sie müssen sich jedoch nicht selbst mit der komplexen Erstellung einer solchen Funktionalität befassen, indem Sie einfach die Search API für die geografische Umgebungssuche[7] nutzen. Der grundsätzliche Arbeitsablauf ähnelt sehr dem oben beschriebenen Prozess für die Volltextsuche. Fügen Sie einfach den geografischen Ort als eine der Dokumenteigenschaften hinzu und nehmen Sie dann die geografische Umgebungsbedingung in Ihre Suchabfragen auf.

Benutzerdefinierte Wartung und andere geplante Aufträge

Neben der synchronen Verarbeitung von Anfragen der mobilen Clients über Google Cloud Endpoints und der asynchronen Nutzung der Aufgabenwarteschlangen hat Ihr Back-End-Code möglicherweise noch einige Verarbeitungsaufgaben, die in verschiedenen regelmäßigen oder halbregelmäßigen Intervallen durchzuführen sind. Vielleicht wünschen Sie zum Beispiel, dass Ihre mobile Lösung jeden Morgen Push-Benachrichtigungen mit einem passenden lokalen Angebot an die mobilen Nutzer sendet oder täglich bzw. mehrmals am Tag diverse anwendungsspezifische Wartungsaufträge durchführt. Vielleicht bietet Ihre Anwendung auch Premium-Abonnements an. Wenn die Abonnements dann ablaufen oder demnächst ablaufen, könnten Sie verschiedene entsprechende Verarbeitungsaufgaben anstoßen, zum Beispiel mithilfe der Mail API eine E-Mail senden, Nutzerkonten deaktivieren usw. Vielleicht wünschen Sie auch, dass Ihr mobiles Back-End alle 10 Minuten einige in Memcache zwischengespeicherte Daten oder jede Stunde die Statistiken der besten Spieler aktualisiert.

Wenn Ihr mobiler Back-End-Code in Google App Engine ausgeführt wird, ist die Konfiguration solcher geplanten Aufträge mit dem Cron-Dienst problemlos möglich. Sie können den Dienst so einrichten, dass einige URLs in Ihrem mobilen Back-End zu den angeforderten Intervallen oder Zeiten aufgerufen werden.

Anwendungsprotokolle speichern und analysieren

Anwendungsprotokolle können viele Informationen darüber liefern, wie die Nutzer Ihre mobile Lösung einsetzen und wie sich Ihr mobiles Back-End verhält. Wenn Ihr mobiler Back-End-Code in Google App Engine ausgeführt wird, können Sie die Anwendungsprotokolle mithilfe der Logs API abrufen. Sie können diese anschließend in Google Cloud Storage hochladen und in Google BigQuery aufnehmen. Wenn es Ihre mobile Clientanwendung zulässt, dass die Nutzer viele Interaktionen innerhalb der Anwendungen durchführen, ohne regelmäßige Anfragen an das Back-End zu senden, empfiehlt es sich, die Back-End-Protokolle mit Protokollen zu erweitern, die die Clientanwendung sammeln und regelmäßig mithilfe der über Google Cloud Endpoints freigeschalteten API an Ihr mobiles Back-End senden kann.

Code außerhalb der App Engine-Sandbox ausführen

Wenn Sie Ihr komplettes mobiles Back-End unter Verwendung von Laufzeitumgebungen erstellen, die von Google App Engine[8] unterstützt werden, müssen Sie sich nicht mit Infrastrukturaspekten wie automatischer Skalierung, Lastenausgleich, Ausfallsicherung usw. befassen. Es bestehen allerdings möglicherweise Anforderungen, die über die Ausführung in der App Engine-Sandbox nicht erfüllt werden können. So könnten Sie zum Beispiel im Rahmen der asynchronen Anfrageverarbeitung oder Batch-Verarbeitung die Nutzung von C/C++ Bibliotheken, benutzerdefinierten Binärdateien oder Perl-Programmen wünschen.

In solchen Szenarien können Sie Google Compute Engine[9] für die Ausführung der Teile Ihres mobilen Back-Ends verwenden, die sich nicht für das Hosting in Google App Engine eignen.

Wenn in Compute Engine ausgeführter Code für die Batch-Verarbeitung oder asynchrone Anfrageverarbeitung verwendet wird[10], können mithilfe des in Google App Engine ausgeführten Codes Aufgaben in eine Pull-Warteschlange (oder mehrere Warteschlangen) gestellt werden. Mithilfe des in Google Compute Engine ausgeführten Codes können diese Aufgaben dann über die Task Queue REST API[11] angefordert werden.

Andere relevante Dienste und APIs

Google bietet noch viele andere Dienste und APIs für den Einsatz in mobilen Lösungen an. Dazu gehören: AdMob, Analytics und verschiedene Maps APIs wie Places API, Android Maps und iOS Maps SDKs. Die Nutzung dieser Dienste und APIs wird im vorliegenden Dokument nicht behandelt.

Fazit

Mithilfe der Google Cloud Platform können Sie problemlos ein Back-End für Ihre mobile Lösung erstellen. Sie müssen sich dabei nicht um Installationen und Infrastruktur kümmern. Ihre Lösung kann hoch skalierbar ausgelegt werden und die Voraussetzungen für die Erstellung überzeugender Anwendungen erfüllen. Unter anderem können Sie die integrierte Nutzerauthentifizierung nutzen, native Android- und iOS-Anwendungen unterstützen, Daten außerhalb von Mobilgeräten speichern, abrufen und verarbeiten sowie Push-Benachrichtigungen orchestrieren und die geografische Umgebungssuche verwenden.

Beispielanwendung

Sie können die Beispielanwendung Mobile Shopping Assistant (App Engine-Back-End, Android-Client und iOS-Client) herunterladen und das entsprechende Tutorial lesen, um festzustellen, wie die wesentlichen Bestandteile der Architektur, die in diesem Dokument vorgestellt wurden, für ein spezielles Szenario implementiert wurden. Die Beispielanwendung beinhaltet die durch Google Cloud Endpoints freigeschaltete REST API und bietet integrierte OAuth-basierte Clientauthentifizierung, geografische Umgebungssuche, Push-Benachrichtigung, asynchrone Anfrageverarbeitung, geplante Aufgaben und die Bereitstellung von Bildern aus Google Cloud Storage.


Hinweise

[1] In diesem Dokument bezieht sich der Begriff Back-End allgemein auf den Teil einer mobilen Lösung, der außerhalb von Clientgeräten ausgeführt wird. Er ist nicht mit Back-End-Instanzen von Google App Engine zu verwechseln, einem Instanztyp, der von App Engine-Standardinstanzen abweichende Eigenschaften aufweist. Die Google App Engine-Standardinstanzen sollten für viele mobile Lösungen ausreichen. Wenn Sie allerdings spezifische Anforderungen haben, können Sie unter Verwendung eines "Back-Ends" eine mobile Lösung aus einer beliebigen Kombination von App Engine-Standardinstanzen und Back-End-Instanzen zusammenstellen.

[2] Stand Februar 2013: Google Cloud Endpoints ist eine experimentelle Funktion.

[3] In Abschnitt 4.2 der Google App Engine-Nutzungsbedingungen werden Ihre Verpflichtung zum Datenschutz und die Rechte der Endnutzer beschrieben, einschließlich der Anforderung einer rechtlich angemessenen Datenschutzerklärung in Bezug auf die von Ihnen erfassten Daten und der Einholung der notwendigen Einwilligungserklärungen seitens der Endnutzer hinsichtlich der Erfassung, Verwendung, Überwachung und Offenlegung dieser Daten.

[4] Wenn Sie die Verwendung relationaler SQL-Datenbanken vorziehen, können Sie Google Cloud SQL entweder anstelle des Datenspeichers oder zusätzlich zum Datenspeicher nutzen.

[5] Stand Anfang Januar 2013: Sockets API ist eine Funktion, die ausschließlich Trusted Testern zur Verfügung steht, und es wird dafür kein Produktionssupport angeboten.

[6] Stand Anfang Januar 2013: App Engine Search API ist eine experimentelle Funktion mit begrenzten kostenlosen Kontingenten.

[7] Stand Dezember 2012: Die geografische Umgebungssuche funktioniert in Anwendungen, die für App Engine bereitgestellt werden. Sie funktioniert jedoch nicht auf dem lokalen Dev-Server.

[8] Stand Dezember 2012: App Engine unterstützt Java 5 und 6, Python  2.7.3 und 2.5.2 sowie die Go-Laufzeitumgebung (letztere als experimentelle Funktion).

[9] Stand Dezember 2012: Google Compute Engine steht ausgewählten Nutzern als eingeschränkte Demoversion zur Verfügung.

[10] Google Compute Engine kann auch in anderen mobilen Lösungsszenarien verwendet werden, aber eine solche Verwendung wird im Rahmen dieses Dokuments nicht behandelt.

[11] Stand Dezember 2012: Task Queue REST API ist eine experimentelle Funktion.
