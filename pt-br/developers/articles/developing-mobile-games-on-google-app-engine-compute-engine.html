Mobile Articles & Mobile Development Guides — Google Cloud Platform

Empresas que utilizam o Google Cloud Platform

Apresentação

O mercado de jogos para dispositivos móveis está crescendo rapidamente. Espera-se que se torne um negócio de US$16 bilhões em 2016[1]. No quarto trimestre de 2012, o gasto global em jogos para as plataformas iOS da Apple e Android do Google, na verdade, ultrapassou os gastos em jogos para dispositivos móveis dedicados[2].

O modelo de receita dos jogos para dispositivos móveis está mudando de jogos pagos para jogos gratuitos, em que a receita é gerada pelos produtos virtuais, outras compras realizadas no próprio jogo e também pela publicidade dentro dele. Com essas novas tendências, existe uma demanda por jogos para dispositivos móveis mais envolventes e imersivos.

Esses jogos não precisam apenas de experiências do usuário atrativas no cliente, mas também de serviços de back-end avançados e escaláveis. O Google Cloud Platform permite criar facilmente o back-end de jogos para dispositivos móveis, como mostrado neste artigo sobre a solução.

Escopo

Este artigo é destinado a arquitetos e desenvolvedores interessados em criar seus próprios back-ends de jogos para dispositivos móveis e projetar os padrões de interação entre os clientes de dispositivos móveis e o back-end.

A abordagem e a arquitetura apresentadas aqui são aplicáveis a um novo desenvolvimento de um jogo para dispositivos móveis típico. Porém, também há duas outras alternativas que podem ser relevantes, dependendo do cenário e dos requisitos. A primeira é o Mobile Backend Starter, que pode ser considerado caso você prefira começar sem precisar desenvolver seu próprio back-end de jogos para dispositivos móveis personalizado. A segunda alternativa é a arquitetura de Solução de jogos com servidores dedicados, que pode ser considerada caso seu jogo para dispositivos móveis exija o uso de servidores dedicados para o jogo.

Este artigo aborda o desenvolvimento de novos jogos para dispositivos móveis altamente escaláveis no Google Cloud Platform. Ele aplica a arquitetura de referência de soluções para dispositivos móveis publicada aqui e também fornece mais detalhes e exemplos, específicos para a criação de jogos para dispositivos móveis. O artigo acompanha um aplicativo de jogo para dispositivos móveis de exemplo com o código fonte completo.

A criação da parte do cliente dos jogos para dispositivos móveis não é abordada neste documento. No entanto, ele inclui algumas informações básicas relevantes para a comunicação com o back-end de jogos para dispositivos móveis, a autenticação de usuários e o envio de notificações de push.

Requisitos

Os requisitos típicos de um jogo para dispositivos móveis atrativo incluem:

  1. Recursos do jogo:
    • Conteúdo de jogo dinâmico
    • Dados do jogo armazenados fora dos dispositivos
    • Placar
    • Modo multijogador
    • Combinação de jogadores
    • Geração de tabuleiros
    • Bots do site do servidor
  2. Recursos da plataforma:
    • Suporte para dispositivos Android e iOS por meio de aplicativos cliente nativos
    • Notificações push para dispositivos Android e iOS
    • Autenticação do usuário
    • Alta escalabilidade
  3. Recursos comerciais:
    • Compras no jogo
    • Análise do jogo

A criação do seu jogo para dispositivos móveis no Google Cloud Platform pode ajudá-lo a atender a esses requisitos.

Os componentes chave na arquitetura da solução proposta são:

  • Seu cliente do jogo para dispositivos móveis: qualquer combinação de aplicativos para Android, iOS e HTML/JavaScript.
  • Seu aplicativo de back-end de jogos para dispositivos móveis.
  • A camada de comunicação entre seus clientes do jogo para dispositivos móveis e seu back-end para dispositivos móveis.
  • Diversos serviços em nuvem para armazenar dados e imagens do jogo, entregar notificações push etc.

As funções dos componentes chave

Uma das opções de design é determinar quais funcionalidades do seu jogo para dispositivos móveis serão executadas por cada componente da solução. Em especial, qual funcionalidade será implementada no seu aplicativo cliente e qual será implementada pelo back-end de jogos para dispositivos móveis e fornecida pelos serviços em nuvem.

Com smartphones e tablets cada vez mais potentes, é conveniente implementar o máximo de funcionalidades possível nos clientes de jogos. A função principal do back-end de jogos para dispositivos móveis é aprimorar a experiência geral, torná-la mais envolvente e imersiva, além de fornecer funcionalidades que não seriam possíveis apenas com o aplicativo cliente em si.

Muitos jogos para dispositivos móveis populares usam os back-ends para dispositivos móveis para:

  • Fornecer conteúdo de jogo dinâmico
  • Facilitar os jogos do modo multijogador, incluindo a combinação de jogadores
  • Armazenar dados fora dos dispositivos móveis, por exemplo, o último nível concluído ou as pontuações atingidas nas jogadas anteriores
  • Gerenciar o placar
  • Orquestrar as notificações push
  • Facilitar as compras no jogo
  • Fornecer análises do jogo

Back-end[3] de jogos para dispositivos móveis na nuvem

Quando o Google Cloud Platform é usado para ativar seu jogo para dispositivos móveis, é possível executar o back-end de jogos no Google App Engine, uma plataforma usada por muitos jogos on-line de sucesso desenvolvidos por empresas grandes e pequenas. Ele oferece estes benefícios da infraestrutura do Google:

  • Escalabilidade automática
  • Balanceamento de carga automático entre as instâncias do back-end para dispositivos móveis
  • Fornecimento de conteúdo estático e armazenamento em cache
  • Confiabilidade, desempenho e recursos de segurança
  • Monitoramento e gerenciamento
  • Proteção contra DoS configurável
  • Integração com as Contas do Google para a autenticação OAuth2 (para clientes Android e iOS)

Desenvolvimento de um jogo para dispositivos móveis no Google Cloud Platform

Cenário do jogo de exemplo

A arquitetura discutida neste artigo pode ser aplicada a muitos jogos para dispositivos móveis. O cenário de exemplo apresentado a seguir é usado principalmente para deixar o design mais específico. A maioria das opções de design discutidas também é relevante para outros jogos para dispositivos móveis.

Griddler: jogo de charadas com modo multijogador para dispositivos móveis

Imagine que você deseja criar um jogo de charadas ou trivialidades com modo multijogador para dispositivos móveis chamado Griddler. Dois ou mais jogadores competem para resolver o máximo de charadas possível em um período de tempo específico. Essa competição será chamada de partida. Todos os jogadores em uma partida visualizam a mesma grade de letras. Por exemplo, uma como esta:

Todos os jogadores precisam descobrir as respostas e realizar um gesto de toque para formar a palavra correta usando as letras adjacentes na grade, como mostrado na Figura 4.

Quando um jogador descobre a palavra correta ou decide pular uma determinada charada, ele é desafiado com outra charada para a mesma grade, como mostrado na Figura 5.

Os clientes do jogo recuperam a definição do tabuleiro do back-end de jogos para dispositivos móveis no início de cada partida. Dessa forma, é possível adicionar novos tabuleiros e charadas sem que seja necessário atualizar os aplicativos clientes. A sequência de charadas de um determinado tabuleiro pode ser aleatória ou fixa. A definição do tabuleiro recuperada inclui as charadas para ele e o tempo estabelecido para uma determinada partida. Quando o tempo estabelecido chega ao fim, o jogador que resolveu mais charadas vence. No caso de empate, o jogador que resolveu as charadas mais rápido vence.

A qualquer momento, podem haver milhões de partidas em andamento, cada uma delas sendo jogada por 1 a N jogadores. Além disso, o Griddler possui o modo de jogador único, que pode ser usado para treinar a mecânica do jogo ou apenas resolver charadas por diversão.

Além de fornecer as definições de tabuleiro para os clientes, as responsabilidades do back-end de jogos para dispositivos móveis incluem:

  • Gerenciar a lista de jogadores e suas estatísticas (por exemplo, de quantas partidas um jogador participou em diferentes dispositivos, quantas partidas do modo multijogador ele venceu).
  • Gerenciar a lista de dispositivos usados pelos jogadores para poder enviar notificações push para esses dispositivos.
  • Enviar convites para partidas do modo multijogador.
  • Retransmitir a aceitação, a recusa ou o cancelamento do convite.

Componentes chave das soluções de jogos para dispositivos móveis

É possível implementar facilmente o Griddler e outros jogos para dispositivos móveis na arquitetura discutida neste artigo. Os clientes podem ser implementados como aplicativos nativos para Android ou iOS. O back-end de jogos para dispositivos móveis pode ser executado no Google App Engine. É possível usar o Google Cloud Endpoints para expor a API do back-end para dispositivos móveis e consumir essa API dos clientes.

Os aplicativos cliente de jogos para dispositivos móveis

Os aplicativos cliente Android e iOS são responsáveis por todas as interações e interface do usuário do jogo, assim como pela mecânica do jogo. O design dessa funcionalidade não é muito diferente do usado em um jogo para dispositivos móveis que não utiliza back-ends de jogos para dispositivos móveis e, dessa forma, está fora do escopo deste artigo.

A diferença é que os clientes Griddler utilizam a tecnologia de back-end de jogos para dispositivos móveis. O cliente se comunica com o back-end para facilitar as partidas do modo multijogador e a combinação de jogadores. Isso também inclui lidar com as notificações push recebidas com os convites de partidas enviados pelo back-end. Além disso, o cliente também recupera do back-end todo o conteúdo dinâmico do jogo, como a definição do tabuleiro de uma determinada partida. Esse conteúdo inclui ainda a grade de letras e todas as charadas de uma grade específica, assim como as respostas válidas para essas charadas.

Após o início da partida, seja uma partida do modo multijogador ou de jogador único, o cliente não precisa se comunicar com o back-end até a partida terminar. Depois, o cliente envia os resultados da partida para o back-end de jogos para dispositivos móveis e ele armazena esses resultados e determina quem venceu a partida do modo multijogador.

O back-end de jogos para dispositivos móveis

O back-end de jogos para dispositivos móveis precisa expor a API adequada para o cliente para ativar a mecânica e a experiência do usuário desejadas.

A Griddler API inclui métodos para:

  • Gerenciamento da partida, por exemplo, startNewSinglePlayerGame que, entre outras coisas, recupera a definição de um tabuleiro, e gameFinished.
  • Convites de jogador, por exemplo, sendInvitation, acceptInvitation etc.
  • Notificações, por exemplo, registerDeviceForPushNotification.

É possível expor esses métodos para os clientes conforme descrito na próxima seção.

Ao lidar com solicitações de API síncronas, o back-end para dispositivos móveis armazena, recupera e gerencia dados em um armazenamento persistente. Além disso, para dados acessados com frequência, como as definições de tabuleiro, ele também usa o cache de memória.

Além de lidar com solicitações do cliente, o back-end para dispositivos móveis é responsável pelo processamento assíncrono necessário para o jogo para dispositivos móveis, pela orquestração de notificações push, por lidar com tarefas agendadas e assim por diante.

Comunicação entre os clientes do jogo para dispositivos móveis e o back-end de jogos para dispositivos móveis

Conforme discutido no artigo sobre soluções para dispositivos móveis, a forma recomendada de expor a API do back-end para dispositivos móveis e consumi-la no cliente para dispositivos móveis é usando o Google Cloud Endpoints[4]. Essa tecnologia permite definir a API com base em anotações simples no código fonte e usar ferramentas para gerar bibliotecas cliente para Android e iOS fortemente tipificadas, bem como códigos JavaScript para interagir com a API nos clientes da Web. Ela também oferece suporte às autenticações baseadas em OAuth2 com Contas do Google, para que o código do back-end de jogos para dispositivos móveis possa saber a identidade do jogador.

No Griddler, por exemplo, o método para enviar um convite é declarado desta forma:

@ApiMethod(httpMethod = "PUT", path ="game/{gameId}/invitation/{playerId}")
public InvitationResult sendInvitation(@Named("gameId") Long gameId,
 @Named("playerId") Long playerId, User user) throws ServiceException { }

O código acima declara a sendInvitation API que, na transmissão, será representada como uma solicitação HTTP PUT usando o URL especificado no argumento de caminho. Os argumentos gameId e playerId serão passados explicitamente pelo cliente. O objeto de usuário será preenchido pela infraestrutura do Google Cloud Endpoints com base nas informações de autenticação fornecidas de maneira implícita pelo cliente. Consulte a documentação do Cloud Endpoints para encontrar mais informações sobre a anotação @API.

Quando a ferramenta Cloud Endpoints for usada para gerar as bibliotecas cliente, o cliente Android poderá invocar essa API exposta do back-end para dispositivos móveis apenas chamando:

InvitationResult invitationResult = gameBackend.sendInvitation(
gameId, playerId)).execute();

Normalmente, é preciso executar essa chamada de maneira assíncrona para evitar o bloqueio das chamadas de rede no thread de IU (interface de usuário).

Armazenamento de dados[5]

Normalmente, jogos para dispositivos móveis envolventes e imersivos precisam armazenar dados fora dos dispositivos do cliente.

Alguns jogos para dispositivos móveis podem usar conteúdo dinâmico representado como objetos grandes e, normalmente, binários. Esse conteúdo inclui imagens que definem partes do mundo do jogo virtual ou avatares enviados por upload pelos jogadores. Para esse tipo de dado, é possível usar o Google Cloud Storage, um serviço que pode armazenar objetos e arquivos com até terabytes de tamanho. Consulte também a seção "Fornecimento e processamento de imagens" abaixo.

Os outros tipos de dados que precisam ser armazenados fora dos dispositivos do cliente são entidades e propriedades específicas. Essas propriedades e entidades podem incluir informações sobre o nível mais alto concluído em um jogo para dispositivos móveis e outras informações sobre os jogadores, como apelidos, registro do dispositivo dos jogadores para notificações push, registros das compras no jogo e outros dados específicos do jogo.

As propriedades dessas entidades também podem incluir uma referência a objetos armazenados no Google Cloud Storage, como o nome do objeto e, opcionalmente, um URL ou nome de intervalo.

Com o back-end para dispositivos móveis hospedado no Google App Engine, o Cloud Datastore é o local recomendado para o armazenamento dessas entidades. Ele oferece um armazenamento de dados de objetos NoSQL,[6] altamente escalável e sem esquema, com transações atômicas e um mecanismo de consulta.

No Griddler, por exemplo, as entidades chave são:

  • Jogador
  • Tabuleiro: a definição da grade de uma partida e as charadas dessa grade.
  • Partida: uma instância de um ou mais jogadores que competem para resolver charadas em um determinado Tabuleiro.
  • Convite: uma entidade usada para iniciar jogos no modo multijogador
  • GamePlay: uma entidade que representa o estado e os resultados de uma Partida disputada por um Jogador. Para jogos no modo multijogador, há várias entidades GamePlay para uma Partida específica (uma por Jogador).
  • Dispositivo: informações específicas da plataforma necessárias para enviar notificações push para um determinado dispositivo, por exemplo, token do dispositivo para iOS e ID de registro de GCM para Android).

Para oferecer suporte para forte consistência, Convite e GamePlay pertencem ao grupo de entidades originado na entidade ancestral Partida. De maneira semelhante, a entidade Dispositivo pertence ao grupo de entidades originado na entidade ancestral Jogador.

Ao usar o Datastore para armazenar as entidades, é possível escalonar os jogos para dispositivos móveis facilmente para milhões de jogadores.

Como o Griddler não usa dados grandes ou binários na implementação atual, ele armazena seus dados totalmente no Datastore. No entanto, se forem implementadas melhorias para fornecer dicas visuais para as charadas ou para permitir que os jogadores façam upload dos seus avatares, o Griddler poderá usar o Google Cloud Storage para armazenar as imagens.

Otimização do acesso aos dados com o Memcache

Quando a carga aumenta, o back-end de jogos para dispositivos móveis pode ser composto por milhares de instâncias individuais que lidam com as solicitações. Se os clientes de jogos para dispositivos móveis estiverem realizando solicitações frequentes para o back-end para recuperar os mesmos dados, é possível otimizar o acesso aos dados para evitar recuperar os dados do armazenamento persistente para todas as solicitações.

Um back-end executado no Google App Engine pode usar um serviço distribuído de cache de dados na memória chamado Memcache para armazenar e recuperar os dados acessados com frequência (Fig. 9). O uso do Memcache não apenas permite que o back-end de jogos para dispositivos móveis apresente um desempenho melhor e seja melhor escalonado, mas também reduz o custo de acessar o Armazenamento de dados do App Engine, pois o aplicativo enviará menos solicitações para ele.

Por exemplo, podem haver milhões de jogadores jogando Griddler ao mesmo tempo e, potencialmente, haver até centenas de milhares de jogadores resolvendo charadas do mesmo tabuleiro (grade de letras) simultaneamente. Em vez de sempre recuperar as definições do Tabuleiro, faz sentido armazená-las em cache.

Processamento assíncrono com filas de tarefas

Às vezes, não é possível ou desejável realizar todo o trabalho associado a uma solicitação recebida de um cliente do jogo para dispositivos móveis antes do envio da resposta de volta para o dispositivo do usuário.

Por exemplo, para tornar uma partida mais envolvente e social, o Griddler poderia ter um recurso para notificar os amigos do jogador quando ele atingisse uma pontuação muito alta e alcançasse a liderança. Ele poderia ser acionado quando o cliente chamasse a gameFinished API. No entanto, é importante considerar que o jogador pode ter muitos amigos e que o cliente não deve ficar bloqueado até que todos os amigos sejam notificados. Esse design síncrono não apenas bloquearia o cliente, mas também não seria confiável. Em vez disso, como parte da gameFinished API, o back-end de jogos para dispositivos móveis pode armazenar os resultados da partida, registrar a necessidade de processar as notificações e, imediatamente, enviar a resposta para o cliente sem esperar até que todas as notificações sejam processadas.

O código executado no Google App Engine pode usar um serviço chamado Filas de tarefas para enfileirar informações sobre o trabalho (tarefas) que precisa ser realizado de maneira assíncrona. No exemplo discutido anteriormente, o aplicativo poderia enfileirar uma tarefa para notificar os amigos do jogador de que ele chegou às 20 maiores pontuações.

Normalmente, você poderia configurar o serviço de fila de tarefas para enviar as tarefas enfileiradas (modelo de push) para o código executado no Google App Engine como solicitações HTTP para um URL selecionado. O back-end de jogos para dispositivos móveis pode processar essas solicitações por meio de gerenciadores de solicitações da Web regulares. No exemplo acima, o gerenciador de solicitações poderia recuperar as informações da tarefa da solicitação HTTP, pesquisar os amigos do jogador e notificar cada um deles, criando postagens em um site de mídia social em nome do jogador ou enviando notificações push conforme descrito abaixo.

De maneira alternativa, o código do back-end para dispositivos móveis pode puxar as tarefas das filas. Isso pode ser útil, por exemplo, se você desejar controlar explicitamente o processamento das tarefas. Outro cenário em que puxar as tarefas é útil é quando você deseja processar as tarefas fora do App Engine, por exemplo, pelas instâncias do Google Compute Engine, conforme discutido no artigo sobre Soluções paradispositivos móveis.

Notificações push

As notificações push permitem que os back-ends de jogos notifiquem os clientes sobre alterações de estado e outras informações relevantes sem a necessidade de uma verificação constante. É possível usar as notificações push para criar uma experiência do usuário avançada, enquanto há otimização do uso da largura de banda e da bateria. Dependendo da plataforma do cliente, o jogo para dispositivos móveis pode usar essa tecnologia para exibir "notificações do sistema" e outras notificações nos dispositivos dos usuários, mesmo se o cliente do jogo para dispositivos móveis não estiver ativamente em execução. Também é possível usar esse recurso para continuar a fornecer os dados relevantes para o cliente de jogos para dispositivos móveis.

No Griddler, as notificações push são usadas para enviar convites quando um jogador deseja desafiar um amigo. O uso de notificações push permite convidar amigos que não estejam com o jogo ativamente em execução no seu dispositivo.

Com o código do back-end de jogos para dispositivos móveis executado no Google App Engine, é possível orquestrar o envio de notificações push para os usuários utilizando estas tecnologias:

  • Google Cloud Messaging para Android (GCM): um serviço do Google com o qual é possível enviar dados para usuários que tenham seu aplicativo cliente de jogo para dispositivos móveis instalado nos dispositivos com Android deles. Se você usar o Eclipse, o esqueleto de código poderá ser gerado pelo Plug-in do Google para o Eclipse.
  • Sockets API[7] para enviar notificações push usando o Serviço de Notificação Push da Apple® para os usuários do iOS. Para começar, leia o artigo sobre Osquestração de notificações push para iOS no Google Cloud Platform e revise o código de exemplo publicado no Github.

Fornecimento e processamento de imagens

Algumas imagens estáticas, como os diversos ícones que fazem parte do jogo e são usados para criar a IU do aplicativo cliente, podem ser empacotadas, normalmente em diversas resoluções, nos clientes de jogo para Android e iOS nativos. No entanto, isso aumenta o tamanho do pacote. Se o pacote for grande, poderá desencorajar os usuários a fazerem o download do aplicativo ou deixá-los mais propensos a desinstalá-lo quando ficarem sem espaço nos seus dispositivos.

Normalmente, não é conveniente ou possível empacotar todas as imagens de que um jogo precisa em um aplicativo cliente. Por exemplo, seu jogo para dispositivos móveis pode ter mundos virtuais grandes e/ou dinâmicos com elementos gráficos elaborados que são representados como imagens com tamanho total de centenas de megabytes ou gigabytes. Mesmo se o tamanho das imagens não for tão grande, será melhor adicionar novos elementos gráficos ao jogo sem fazer com que os usuários precisem atualizar seus aplicativos cliente.

Por exemplo, o Griddler poderia ser expandido para incluir alguma dica visual para uma determinada charada. Poderia também permitir que os usuários fizessem upload de imagens representativas dos seus avatares.

Se for possível preparar essas imagens em resoluções e formatos otimizados para seus clientes de jogo para dispositivos móveis, então será possível fazer upload delas para o Google Cloud Storage e fornecer URLs diretos para essas imagens aos clientes. Essa abordagem pode ser muito bem escalonada, uma vez que permite que os clientes para dispositivos móveis recuperem as imagens diretamente do armazenamento em nuvem escalável, sem envolver o código do back-end para dispositivos móveis.

Em alguns casos, é possível manipular e processar imagens, além de armazená-las. Por exemplo, forneça as imagens em tamanhos ajustados dinamicamente de acordo com a resolução dos dispositivos do cliente. Nesses casos, use o Serviço de imagens do Google App Engine para redimensionar, girar, virar e cortar imagens, assim como melhorar fotografias usando um algoritmo pré-definido. É possível realizar essas operações em imagens do Google Cloud Storage, além de outras fontes. As imagens modificadas podem ser fornecidas aos clientes diretamente da infraestrutura dinâmica de fornecimento de imagens disponibilizada pelo Google, sem envolver o código do back-end de jogos para dispositivos móveis. O código é responsável apenas por usar a Images Service API para receber o URL correto da imagem e por fornecer esse URL aos clientes de jogos para dispositivos móveis.

Tarefas agendadas

Além de processar solicitações dos clientes de jogo para dispositivos móveis de maneira síncrona por meio do Google Cloud Endpoints e de maneira assíncrona usando as filas de tarefas, o código do back-end de jogos para dispositivos móveis pode precisar realizar processamentos em vários intervalos regulares ou semirregulares. Por exemplo, você deseja que o back-end de jogos para dispositivos móveis envie notificações push aos jogadores que não usam seu jogo há um determinado número de dias e incentivá-los a jogar novamente e deseja realizar esse processamento em momentos específicos do dia.

O back-end de jogos para dispositivos móveis também poderia atualizar alguns dados armazenados em cache no Memcache a cada cinco minutos ou atualizar as estatísticas dos principais jogadores a cada 15 minutos.

Com o código do back-end de jogos para dispositivos móveis executado no Google App Engine, é fácil configurar essas tarefas agendadas usando o Cron Service. É possível configurá-lo para invocar determinados URLs no back-end para dispositivos móveis nos intervalos solicitados ou momentos específicos.

Serviços relacionados a jogos do Google Play

Uma maneira fácil de tornar seus jogos para Android, iOS e Web mais sociais é usar os Serviços relacionados a jogos do Google Play. É possível premiar os jogadores com conquistas, orientar o envolvimento com placares e tornar o jogo mais dinâmico ao permitir que vários jogadores compitam ou cooperem simultaneamente.

Aplicativos de exemplo

É possível fazer o download do aplicativo de exemplo Griddler no Github. O back-end para dispositivos móveis é implementado em Java e expõe a REST API usando o Cloud Endpoints. Ele armazena os dados no Datastore usando uma API de nível inferior e envia notificações push. Os clientes de jogo para Android e iOS são implementados como aplicativos nativos. A comunicação entre os clientes e o back-end de jogos para dispositivos móveis é autenticada usando o OAuth e as Contas do Google.

Também é possível fazer o download do Cloud Adventure, outro exemplo de jogo para dispositivos móveis desenvolvido em uma arquitetura semelhante. O Cloud Adventure é um jogo para Android baseado em texto que mostra alguns cenários interessantes, como uma sala de espera do jogo para vários usuários e seleção de apelidos únicos. Ele também demonstra como usar uma combinação do Google Cloud Messaging para Android e o Cloud Endpoints para lidar com o término de uma partida e totalização dos resultados.

Próximas etapas

Caso ainda não tenha feito isso, visite o portal do Google Cloud Platform e clique no link "Teste agora". Você será guiado pelas etapas necessárias para criar um projeto gratuito do Google Cloud, que pode ser usado para implantar o jogo de exemplo Griddler. Depois, você pode brincar com esse exemplo, explorá-lo e tentar estendê-lo, por exemplo, ao adicionar uma dica visual a cada charada.

Conclusão

Ao utilizar o Google Cloud Platform, é possível criar com facilidade um back-end para dispositivos móveis para seus jogos. Você pode se concentrar em desenvolver seu jogo sem precisar se preocupar com a implantação, manutenção e solução de problemas da infraestrutura de TI. Também pode criar jogos para dispositivos móveis imersivos, envolventes e atraentes, que podem ser escalonados perfeitamente de alguns usuários para milhões de jogadores no mundo todo.

Já existem diversos jogos bem-sucedidos que utilizam a tecnologia do Google Cloud Platform e mal podemos esperar para ouvir sobre suas histórias de sucesso.
