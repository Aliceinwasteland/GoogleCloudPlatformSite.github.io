Primeiros passos com o Google BigQuery

Resumo executivo

Este documento destina-se a analistas de dados e profissionais de TI que desejam entender como usar o Google BigQuery no seu ambiente de TI atual. Ele fornece um conhecimento básico de como funciona, uma visão geral do processo completo de carregamento de dados no BigQuery para visualização e práticas de design que devem ser consideradas ao utilizar o BigQuery.

O BigQuery é compatível com padrões e protocolos abertos, como linguagem de consulta semelhante a SQL e uma API baseada em REST. É compatível também com ferramentas tradicionais de Extração, Transformação e Carregamento (ETL, na sigla em inglês) para ingestão de dados e ferramentas de Inteligência Comercial (BI, na sigla em inglês) para visualização de dados, facilitando a integração do BigQuery no ambiente de TI existente de uma organização.

Panorama do Big Data

O panorama da análise de dados tem mudado rapidamente nos últimos anos. Cada vez mais organizações estão usando dados para orientar o processo de tomada de decisões. Ao mesmo tempo, o ritmo da geração de dados está acelerando radicalmente. Essa mudança para Big Data dentro das organizações é impulsionada por muitos fatores. Alguns fatores que influenciam essa tendência são:

  1. A diminuição radical nos custos de armazenamento, que permite às organizações coletar e utilizar grandes quantidades de dados de modo mais econômico do que nunca.
  2. Os avanços tecnológicos agora permitem a automatização da coleta de dados em cada ponto do processo, o que resultou na captura de conjuntos de dados muito maiores. A combinação de captura de conjuntos de dados tradicionais, por meio de processos como gerenciamento de pedidos, e processos automatizados impulsionados por tecnologias, como tags RFID, sensores, registros de máquinas, interações de dispositivo móvel, está estimulando um enorme crescimento de dados.
  3. Inovações em tecnologias de análise de dados tornaram possível a análise de conjuntos de dados em larga escala. Tecnologias desenvolvidas pelo Google, como MapReduce, BigTable e Dremel, estão começando a ser usadas no setor. Essas tecnologias permitem que a análise dos dados seja realizada em períodos históricos mais longos e em um nível muito mais granular. Isso, por sua vez, fornece novas percepções de negócios que não eram possíveis no passado.

Essa mudança no panorama, no entanto, também apresenta um novo conjunto de desafios. Muitas empresas estão começando a enfrentar problemas como encontrar uma infraestrutura escalável e os recursos computacionais necessários para armazenamento, processamento e análise de grandes conjuntos de dados. A infraestrutura e os processos de TI tradicionais não são suficientes para o processamento e gerenciamento de grandes conjuntos de dados. Por exemplo, com a análise de Big Data, uma empresa geralmente requer muita experimentação antes de desenvolver percepções duráveis. Experiências com dados também significam que é difícil identificar o retorno sobre o investimento (ROI, na sigla em inglês) antecipadamente, o que torna difícil para as organizações de TI planejar a capacidade necessária para gerenciar grandes conjuntos de dados. Devido ao ritmo em que as condições de negócios mudam, os usuários de negócios desejam validar suas hipóteses rapidamente. O tradicional processo de Business Intelligence (BI)/Data Warehousing (DW), com seu longo ciclo de modelagem de dados, arquitetura de dados e planejamento de capacidade, não está satisfazendo suas necessidades.

Google BigQuery

O BigQuery é um serviço de análise de dados totalmente gerenciável oferecido na nuvem, que permite a análise de negócios ágil do ambiente atual de negócios em rápida evolução. Começar a usar o BigQuery não requer despesas de capital adicionais (CAPEX, na sigla em inglês) para comprar e configurar hardware ou software. O serviço BigQuery está prontamente disponível, com um Contrato de nível de serviço de tempo de disponibilidade maior ou igual a 99,9%[1]. Os recursos de armazenamento e computação são ajustados automaticamente de acordo com o tamanho dos dados e os requisitos de análise. O modelo "pague somente pelo que usar" [2], o desempenho "incrivelmente rápido" [3], uma conhecida linguagem familiar semelhante a SQL e a facilidade de aprovisionar dados no BigQuery sem a necessidade de uma arquitetura de dados complexa: tudo isso permite que as organizações executem rapidamente consultas ad-hoc, validem hipóteses e repitam a análise para entender os dados de uma maneira ágil e econômica. Os resultados da consulta são obtidos em segundos e não em horas.

Como o custo e o tempo necessários para a configuração da infraestrutura de análise de dados não são mais barreiras, as equipes de negócios, juntamente com seus desenvolvedores, podem configurar e executar projetos de análise rapidamente. A velocidade resultante permite que a linha de equipes de negócios e diretores de marketing coloque rapidamente em execução soluções analíticas importantes que ajudam a aumentar a receita dos seus negócios.

BigQuery em ação

As empresas têm aproveitado os pontos fortes do BigQuery em custo, desempenho, escalabilidade e disponibilidade de funcionamento em uma ampla gama de casos de uso. Abaixo estão alguns exemplos que ilustram como o BigQuery tem sido utilizado de forma eficaz.

Pesquisa de registros do Gmail [4]: a equipe do Google Apps usa o alto desempenho do BigQuery para incrementar o recurso interativo "Pesquisa de registros do Gmail" no painel de controle do administrador. Eles decidiram usar o BigQuery devido à sua capacidade de analisar conjuntos de dados de vários terabytes com bilhões de cabeçalhos e oferecer resultados precisos em apenas alguns segundos. O recurso "Pesquisa de registros do Gmail" permite que o administrador do aplicativo de domínio responda a perguntas como:

  1. O que aconteceu com uma mensagem de entrada ou de saída?
  2. Uma mensagem foi enviada para meu domínio e marcada como spam?
  3. Que usuário enviou ou recebeu uma mensagem específica?

Otimização das vendas de anúncios on-line no setor de férias: a Crystalloids, uma empresa de análise com sede em Amsterdã, trabalhou com a Auto Camper Service International (ACSI), uma dos principais editoras de guias de acampamento da Europa, para melhorar as vendas de anúncios on-line. Usando o BigQuery, a equipe de vendas de anúncios da ACSI foi capaz de contrastar o número médio de pedidos de reserva entre os acampamentos que fizeram ou não propagandas, dada a mesma região e as mesmas características. Esses dados permitiram vender anúncios de forma mais eficaz.

Gerenciamento de reservas e rotas em viagens [5]: a redBus.in, uma agência de viagens on-line que introduziu a venda de bilhetes de ônibus pela Internet na Índia, escolheu o BigQuery para analisar dados de reservas e inventário do seu sistema de centenas de operadoras de ônibus que atendam a mais de 10.000 rotas. Eles foram capazes de analisar conjuntos de dados enormes de 2 terabytes em menos de 30 segundos e gastaram 80% menos do que gastariam em uma infraestrutura Hadoop [6].

Como o BigQuery funciona

O BigQuery é baseado em Dremel [7], uma tecnologia pioneira do Google, e muito usado dentro do Google. O BigQuery utiliza armazenamento colunar e árvores de execução de diversos níveis para alcançar um desempenho interativo para consultas em bancos de dados de vários terabytes. A implementação técnica é diferente daquela de um sistema de banco de dados relacional (RDBMS, na sigla em inglês) e das soluções tradicionais de armazenamento de dados. O BigQuery é compatível com inserções de dados, exclusão (eliminação de tabelas) e a capacidade de adicionar colunas ao esquema da tabela após os dados já estarem carregados. Ele não é compatível com atualizações diretas nos dados armazenados. Mesmo que atualizações diretas em nível de linha não sejam suportadas, muitos clientes podem realizar atualizações quebrando grandes conjuntos de dados em tabelas finalizadas e não finalizadas e substituindo as tabelas não finalizadas conforme necessário, de modo que a análise final reflita os resultados das atualizações de dados.

A vantagem de desempenho do BigQuery vem da sua arquitetura de processamento paralelo. A consulta é processada por milhares de servidores em uma estrutura de árvore de execução de vários níveis, com os resultados finais agregados na raiz. O BigQuery armazena os dados em um formato de colunas, de modo que apenas os dados das colunas que estão sendo consultadas são lidos. Essa arquitetura tem ganhos significativos de desempenho de E/S sobre o modelo tradicional de banco de dados relacional/armazenamento de dados, pois não é lido todo o registro para cada consulta.

Criação de uma solução de Big Data no BigQuery: pipeline até análise

É fácil criar uma solução completa no BigQuery. Este artigo apresenta várias abordagens para processar e visualizar os dados. Inclui também recomendações de projeto para a utilização do BigQuery de forma eficaz.

Pipeline de dados

O processo de carregamento de dados no BigQuery segue estas etapas:

  1. Extraia dados de origem do sistema de origem.
  2. Desnormalize os dados, pois o BigQuery tem melhor desempenho com menos JOINs (opcional).
  3. Transforme os dados no arquivo de valores separados por vírgulas (CSV, na sigla em inglês) [8] ou no arquivo Javascript Object Notation (JSON).
  4. Faça upload do arquivo CSV ou JSON no Google Cloud Storage. É possível fazer isso usando o utilitário de linha de comando gsutil do Google Cloud Storage. Algumas ferramentas de ETL de terceiros podem fazer upload diretamente no BigQuery.
  5. Carregue o arquivo CSV ou JSON do Google Cloud Storage no BigQuery usando o utilitário de linha de comando bq [9] ou BigQuery REST API.
Uso do Google Cloud Platform para pipeline de dados

O processo de uso do Google Cloud Platform para fazer upload de dados no BigQuery envolve o upload dos arquivos CSV ou JavaScript Object Notation (JSON) para o Google Cloud Storage antes de carregar os dados no BigQuery. O Cloud Storage e o BigQuery fornecem ferramentas de linha de comando de fácil utilização para fazer upload de dados. Como alternativa, também é possível utilizar a REST API para proporcionar uma integração programática no ambiente de computação atual.

É possível executar ferramentas de ETL no Google Compute Engine para transformar os dados de origem em um formato pronto para ser importado para o BigQuery.

A estrutura MapReduce do App Engine e o Hadoop em execução no Google Compute Engine são duas opções para a realização de ETL. O esboço a seguir sugere como um aplicativo do App Engine pode ser usado para orquestrar qualquer abordagem.

  1. Faça upload dos dados de origem para o Google Cloud Storage. Para isso, use a ferramenta de linha de comando gsutil ou a Cloud Storage REST API.
  2. O aplicativo do Google App Engine detecta automaticamente a presença desses arquivos. É possível implementar a detecção automática de algumas formas:
    1. Use o serviço Cron do App Engine para consultar periodicamente o intervalo do Cloud Storage, usando a REST API para detectar objetos com upload recente.
    2. Use a Cloud Storage Bucket Change Notification AP.
      É um modelo publicar/subscrever em que o Cloud Storage age como editor e o aplicativo age como assinante. Na inicialização do aplicativo, um método de retorno de chamada está registrado para receber mensagens quando há arquivos novos ou atualizados.
  3. O aplicativo do App Engine pode invocar um destes pipelines de MapReduce:
    1. Uso da estrutura MapReduce do App Engine
    2. Início de um processo para executar o Hadoop no Google Compute Engine. O aplicativo usa a REST API do Google Compute Engine para configurar as instâncias e iniciar a tarefa de MapReduce. Os arquivos processados são gravados de volta no Cloud Storage.
  4. O aplicativo pode, em seguida, fazer upload dos dados de registro processados no BigQuery usando a REST API.
  5. É possível usar a App Engine Mail API em qualquer parte desse processo para gerar uma notificação de conclusão de tarefa.
Uso de fornecedores de terceiros para pipeline de dados

Basicamente, elas executam todos os passos acima de uma maneira contínua. Essas ferramentas serão especialmente úteis se houver dados da empresa provenientes de diversas fontes. Práticas de projeto

Ao carregar dados no BigQuery, é preciso considerar algumas práticas de projeto desde o início para garantir uma implantação bem-sucedida do projeto de análise de Big Data.

Considere políticas de cota ao arquitetar uma solução

É preciso considerar várias políticas de cota existentes. Um exemplo ocorre com o processamento de dados, em que é importante não exceder o tamanho máximo de arquivo por tarefa de carregamento. Saber dessa limitação é importante porque, se um arquivo exceder o limite, terá que ser dividido em arquivos menores e carregado separadamente. As políticas de cota mais atuais para o BigQuery são documentadas no site de desenvolvedores do Google BigQuery.

Desnormalização de dados antes do carregamento de dados

A implementação técnica do BigQuery é diferente de um RDBMS. O BigQuery usa armazenamento colunar e árvores de execução de vários níveis [10] para alcançar escalonamento e desempenho. O Apêndice 1 explica o processo de desnormalização de tabelas em mais detalhes.

JSON x CSV

O BigQuery é compatível com o carregamento de dados em formato CSV e Java Script Object Notation (JSON). O JSON suporta estrutura aninhada/repetida e representa uma visualização mais orientada a objetos dos dados em comparação com CSV. O JSON também elimina a necessidade de duplicação de dados exigida pelo achatamento de registros em CSV.

O BigQuery fornece funções SQL especiais, FLATTEN e WITHIN, para apoiar a consulta de dados repetidos e aninhados. Nem todas as ferramentas de BI são compatíveis com essas funções especiais do SQL, por isso é uma boa prática testar um pequeno subconjunto de dados com as ferramentas de terceiros desejadas para o formato JSON.

Partição de tabela

O particionamento de uma tabela significa dividi-la em uma ou mais tabelas com o mesmo esquema para fins específicos.

Dados estáticos e variáveis

Como mencionado anteriormente, o BigQuery não é compatível com atualizações de dados que já estão carregados. Ao separar os dados que têm mais chances de mudar daqueles que não têm, as atualizações podem ser feitas simplesmente excluindo uma tabela e preenchendo novamente com os dados atualizados, o que evita a necessidade de recarregar o conjunto de dados inteiro.

Economia

O preço para consultas interativas e em lote é baseado na quantidade de dados processados. É possível otimizar os custos particionando as tabelas de acordo com o padrão de uso. Por exemplo, se a maioria das consultas se concentrar nas tendências mensais, o particionamento da tabela por mês reduzirá os custos da consulta. Se for necessária a consulta de dados em vários meses, ela poderá ser feita por uniões de tabela, como mostrado na Figura 2.

// Find suspicious activity over several months
SELECT FORMAT_UTC_USEC(event.timestamp_in_usec) AS time, request_url
FROM [applogs.events_201205], [applogs.events_201206], [applogs.events_201207]
WHERE event.username = 'root' AND NOT event.source_ip.is_internal;

Figura 2. Exemplo de tabelas particionadas

Se forem necessárias uniões de tabela, não há limite para o número de tabelas que podem ser especificadas, mas há uma limitação da extensão da consulta, que atualmente é de 10 KB.

Armazenamento de dados no Google Cloud Storage como backup

Depois de carregar os dados no BigQuery, considere deixar os dados no Google Cloud Storage como dados no BigQuery.

Operações JOIN

O BigQuery aceita dois tipos de operações JOIN. O uso de cada operação é determinado pelo tamanho da tabela do lado direito do JOIN, enquanto não há nenhum limite de tamanho para a tabela do lado esquerdo. A instrução JOIN básica requer que a tabela do lado direito contenha menos de 8 MB [11] de dados compactados. Essa operação JOIN é projetada para ser rápida, por isso o limite máximo de tamanho para a tabela do lado direito.

Quando ambas as tabelas são extremamente grandes, de dezenas de gigabytes a vários terabytes, é possível usar a operação JOIN EACH. A JOIN EACH faz com que o BigQuery realize uma operação interna que quebra tabelas grandes em partes menores que são processadas com mais eficiência em paralelo. Essa etapa incorre no processamento adicional, ao contrário de uma cláusula JOIN.

Otimização de desempenho usando subselect

Em alguns casos, é possível usar uma instrução subselect para reduzir a quantidade de dados da tabela do lado direito para se encaixar no limite de 8 MB, sem ter de usar JOIN EACH, como mostrado na Figura 3.

// Limit number of fields and rows.
SELECT purchases.amount, users.email_address
FROM PurchaseData AS purchases
AS users
ON purchases.user_id = users.user_id
WHERE purchases.amount > 500

Figura 3. Exemplo de uso de subselect para reduzir o tamanho dos dados

Evite dados extremamente desiguais

Quando JOIN EACH é utilizado e uma resposta de consulta parece estar levando um tempo surpreendentemente longo, é uma indicação de que os dados estão distribuídos de forma desigual. A distribuição desigual faz com que alguns blocos de dados sejam desproporcionalmente maiores do que outros, deixando toda a execução da consulta mais lenta. A Figura 4 pode ser usada para ilustrar esse ponto. Foi projetada para analisar a eficácia de uma campanha de e-mail.

select date_of_visit, page, sum(time_spent_on_page)
from weblog_20130301 w
join each
(select user_id from email_campaign) e
on w.user_id = e.user_id
group by date_of_visit, page

Figura 4. Exemplo de uma consulta que possivelmente contém dados distribuídos de forma irregular

No entanto, se uma grande parte do weblog contiver tráfego de usuários que não estão conectados e o sistema usar um user_id anônimo, essa consulta levará mais tempo do que o necessário para processar os usuários anônimos. Um filtro simples, como mostrado na Figura 5, melhoraria o desempenho de consulta:

select date_of_visit, page, sum(time_spent_on_page)
from weblog_20130301 w
join each
(select user_id from email_campaign) e
on w.user_id = e.user_id
where w.user_id != '<anonymous_user_id>'
group by time_of_visit, page

Figura 5. Exemplo de uso do filtro para eliminar dados que causaram a distribuição irregular de dados

Apêndice 1 - Desnormalização de dados

O BigQuery usa armazenamento colunar e árvores de execução de vários níveis [12] para atingir escalonamento e desempenho para consultas interativas. A implementação técnica é diferente de um RDBMS. O BigQuery tem um melhor desempenho quando o número de junções de tabela é minimizado. Em alguns casos, é preciso desnormalizar as tabelas antes que sejam carregadas no BigQuery.
