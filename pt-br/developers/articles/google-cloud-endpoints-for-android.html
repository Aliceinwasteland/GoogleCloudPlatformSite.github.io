Google Cloud Endpoints para Android

Dicas e práticas recomendadas de um desenvolvedor Java

Apresentação

Você deseja desenvolver um aplicativo para dispositivos móveis de nível internacional. Você usa o Google Cloud Endpoints porque fica sabendo que economizará muito tempo e codificação. Ele é um recurso relativamente novo e você já leu alguns dos excelentes documentos, artigos e tutoriais existentes.

Mesmo com essas informações, você não tem certeza sobre alguns dos pontos mais delicados. Ao tentar gerar sua biblioteca do Cloud Endpoints, você não entende que parte do seu código está acionando o erro. Como você anota seus métodos para não haver conflitos entre os caminhos de REST? Certamente, mapeamentos com código de cores entre o código gerado e as anotações ajudarão.

Encontre as respostas para essas perguntas e muito mais neste artigo. Se você não está preso, mas de qualquer maneira deseja algumas dicas, este ainda é um excelente lugar para começar. Se você é um iniciante e ainda precisa fazer perguntas como: "O que é o Cloud Endpoints?", releia as seções "Visão geral" e Python ou Java. Elas são curtas. Uma revisão rápida será suficiente antes que você precise consultá-las mais tarde durante o desenvolvimento do seu aplicativo.

As seções a seguir abordam algumas dicas gerais, detalhes práticos, problemas que você pode encontrar e soluções para eles. Supõe-se que você já saiba um pouco sobre o Cloud Endpoints, pois este não é um "guia" abrangente, um "tutorial" nem mesmo um "manual". Pense nele mais como um apêndice muito útil para a documentação principal do desenvolvedor. Se não tiver certeza de onde começar, volte e leia a "Visão geral" aqui.

Cloud Endpoints em termos simples

O Cloud Endpoints é um serviço para a criação e exposição de APIs personalizadas com as quais seus aplicativos cliente podem interagir.

Ele existe para facilitar muito sua vida como desenvolvedor. Os engenheiros do Google já se esforçaram muito para escrevê-lo e otimizá-lo para você, libertando-o da preocupação de ter que escrever a maior parte das interações servidor-cliente e protocolos.

Um dos benefícios de poder criar facilmente suas próprias APIs é poder projetá-la da maneira que quiser. O Cloud Endpoints oferece essa flexibilidade, já que permite REST, APIs de estilo RPC ou uma combinação de ambos. Melhor ainda, é possível alterar o design sempre que quiser e implementar essas mudanças rapidamente. Isso é perfeito para um estilo ágil de desenvolvimento.

Um pouco sobre REST básico

Em REST, a idéia básica é expor o estado do sistema como uma coleção de recursos individualmente endereçáveis. Você executa operações por meio da manipulação desses recursos. Um recurso é uma entidade que tem um estado, bem como um nome ou id. Os recursos são armazenados no back-end, seja o Google Cloud Datastore ou outra ferramenta do gênero.

Como regra geral, com o REST, o cliente vê representações dos recursos por meio de solicitações feitas ao servidor. Ele também pode atualizar os recursos, manipulando as representações e enviando as representações atualizadas para o servidor por meio de solicitações subsequentes. É possível representar os recursos de várias formas: XML, JSON, CSV, dentre outros. Neste momento, o Cloud Endpoints é compatível apenas com JSON, que tende a ser menos detalhado e menos vulnerável a erro humano do que XML.

Talvez você já tenha ouvido falar de CRUD (criar, recuperar, atualizar, excluir) ou variantes. É a raiz do REST. Quando você usa o Plug-in do Google para o Eclipse para gerar uma classe Endpoint, esses tipos de métodos básicos são gerados na classe Endpoint.

Você não está limitado a essas ações básicas. Adicionar métodos personalizados oferece muito mais flexibilidade, permitindo realizar operações complexas. Por exemplo, o checkForVictory pode executar várias consultas em um banco de dados, além de uma gravação .

Configuração: uma revisão rápida

Esta seção fornece uma rápida revisão sobre como gerar seu projeto e criar pontos de extremidade no Eclipse. Para mais informações, consulte a documentação do desenvolvedor.

  1. É preciso fazer o download destes itens e configurá-los no seu computador:
    1. Eclipse
    2. Plug-in do Google para sua versão do Eclipse
    3. SDK do App Engine
    4. SDK do Android

    Até o momento, as versões mais recentes foram Eclipse 4.2, SDK do App Engine 1.8.0 e SDK do Android r21.01.1.

  2. Se você ainda não tiver um aplicativo cliente para Android, crie um no Eclipse.
  3. Para criar seu back-end do App Engine, clique com o botão direito do mouse no seu aplicativo para Android e selecione Google > Criar back-end do App Engine.
  4. Para gerar classes Endpoints básicas, clique com o botão direito do mouse nna classe java do seu objeto de dados e selecione Google > Gerar classe do Cloud Endpoints. É possível modificar ainda mais essas classes posteriormente.
  5. Para criar sua biblioteca, clique com o botão direito do mouse no seu aplicativo de back-end do App Engine e selecione Google > Gerar biblioteca do Cloud Endpoints.

Se você for um usuário do IntelliJ, uma outra opção para um ambiente de desenvolvimento Android é o Android Studio no IntelliJ. Este artigo não aborda essa opção, mas você pode ler mais sobre o assunto no site do desenvolvedor Android. Como essa opção é uma pré-visualização no momento, tenha em mente que erros ou alterações podem aparecer de vez em quando. No entanto, ele é extremamente versátil e tem grandes recursos existentes e previstos. Mantenha-se informado, especialmente se você for um usuário ávido de IntelliJ.

Snippets de código de ponto de extremidade

Definição de pontos de extremidade no back-end

É possível agrupar o código do ponto de extremidade do aplicativo App Engine em duas categorias: modelo e API.

O primeiro tipo é seu modelo. Essas classes representam os objetos do seu aplicativo. Em outras palavras, representam os recursos que serão manipulados pela sua API. Por exemplo, caso o universo do seu aplicativo contenha lagostas com bigodes, haverá um objeto de dados como este:

public class Lobster {
  private int whiskerLength;
  public int getWhiskerLength() {
    return whiskerLength;
  }
  public void setWhiskerLength(int whiskerLength) {
    this.whiskerLength = whiskerLength;
  }
}

O segundo tipo é o código de API. A parte de API do código fornece os métodos que podem ser acessados em um aplicativo cliente. Esses métodos manipulam os objetos no seu modelo. Há duas maneiras de começar a escrever o código de API:

  • Clique com o botão direito do mouse em Lobster.java e selecione Google > Gerar classe Endpoint. O Eclipse gera uma nova classe chamada LobsterEndpoint.java. Essa classe será preenchida com anotações e código pré-gerados, prontos para serem chamados a partir do cliente. Esses métodos gerados inserem, atualizam e excluem os objetos Lobster do seu armazenamento de dados. Para modificar ainda mais essa classe, adicione ou remova métodos.
  • Crie sua própria classe Java e a chame como quiser. Contato que esteja anotada corretamente, o Cloud Endpoints a reconhecerá como parte da API. Os métodos de instância privada não farão parte da API.

Um exemplo de uma classe Endpoint:

@Api(name = "lobster")
public class LobsterEndpoint {
  @ApiMethod(name = "lobster.growWhisker", httpMethod = HttpMethod.POST)
  public void growWhisker(Lobster lobster) {
    // grows the lobster’s whisker
    // calculates growth, updates the Lobster in Datastore
  }
  private int calculateWhiskerGrowth(Lobster lobster) {
    // this method is a private utility method
    // it will NOT be visible in your API
  }
}

Dica geral: não fique preso ao código de ponto de extremidade gerado. Ele é útil e contém ótimos padrões de ponto de partida, mas não tenha medo de modificá-lo para seu projeto específico. O código é tão maleável quanto você deseja. Por exemplo, as entidades de DeviceInfo criadas automaticamente usam IDs de registro como chaves. Você ainda pode adicionar novos campos a uma entidade DeviceInfo ou utilizar um campo diferente para a chave.

Consumo de pontos de extremidade pelo cliente

Depois de gerar sua biblioteca do Cloud Endpoints, o novo código que aparece no cliente, geralmente em alguma pasta com "endpoint-libs" no caminho de arquivo, corresponde à funcionalidade fornecida no código do back-end.

A diferença no modelo é que as novas classes estendem a classe GenericJson, de modo que possam ser enviadas como JSON. A classe Lobster recém-criada seria assim:

public final class Lobster extends GenericJson {
  private int whiskerLength;
  public int getWhiskerLength() {
    return whiskerLength;
  }
  public Lobster setWhiskerLength(int whiskerLength) {
    this.whiskerLength = whiskerLength;
    return this;
  }
}

O código cliente da API é uma série de métodos e classes de solicitações JSON, o que facilita muito seu uso em um cliente Android. O resultado da classe LobsterEndpoint.java seria assim:

public Lobster lobster() {...}
public class Lobster {
  public GrowWhisker growWhisker(Lobster lobster) {...}
  public class GrowWhisker extends LobsterRequest {
    private static final String REST_PATH = “growWhisker”
    protected GrowWhisker(Lobster lobster) {...}
    ...
  }
}

É possível consumir esse ponto de extremidade exposto com este código:

service.lobster().growWhisker(lobster).execute();

Se você estiver familiarizado com solicitações HTTP, saberá que, para qualquer caminho REST especificado, você pode fornecer apenas uma definição de POST, uma de GET e assim por diante. Certamente, é possível criar vários POSTs na mesma API de ponto de extremidade, mas seus caminhos REST devem ser diferentes para não haver conflitos. Para encontrar mais informações sobre anotações @ApiMethod, consulte a seção relevante no apêndice.

De maneira ideal, uma vez que a biblioteca é gerada no aplicativo cliente, é preciso isolar o código do cliente gerado. Quaisquer alterações que você queira fazer no modelo ou API devem ser feitas no back-end.

Criação da API

Modelo

  • As classes do seu modelo devem ter determinadas características.
    Se desejar utilizar uma classe na API, estas são as características necessárias:
    • Ser serializável
    • Ter um construtor sem argumentos
    • Implementar getters e setters para seus campos
  • As classes no seu modelo também podem conter outros métodos de instância.
    Lembre-se de que nenhum método que não seja getter ou setter estará do lado do cliente.
  • Como uma definição de classe de objeto no modelo é sinalizada para inclusão no modelo cliente gerado?
    Digamos que você tenha uma classe Clam.java e alguns campos getters e setters (get/setPearl, get/setColor, por exemplo), mas quando você gera a biblioteca do Endpoints, a classe Clam.java não aparece no código gerado. O que aconteceu com ela? O modo de usar o objeto Clam no restante do modelo e no back-end da API determina esse comportamento.
    • Usar a entidade na API forçará o Cloud Endpoints a reconhecer que existem Clams como objetos. 
      @ApiMethod(name = "clam.producePearl", httpMethod = HttpMethod.POST)
public void producePearl(Clam clam) {
  // adds a pearl to the clam
}
    • O uso correto de Clams no seu modelo obriga o Cloud Endpoints a reconhecê-los e gerar um Clam.java do lado do cliente. Para fazer o Cloud Endpoints reconhecer os Clams como objetos, escreva um método setter setClam (Clam clam), que pode ser público, protegido ou privado.
      • Os métodos "setter" devem ter estas propriedades:
        • O nome do método deve começar com "set"
        • O objeto que você deseja definir deve estar nos parâmetros
          Por exemplo:

          setClam(Tree tree) define claramente uma Tree, não um Clam, apesar de o nome do método afirmar o contrário. Esse método gera uma classe Tree.java no cliente, bem como métodos getTree() e setTree(Tree). Não gera um método setClam(Clam).

Sobre os tipos compatíveis

  • Enums. Enums no back-end são gerados como representações de String no código do cliente. Em outras palavras, se você estiver usando a funcionalidade enum, tanto no back-end quanto no cliente, precisará definir o enum em ambos os lugares.
  • Booleano: muitas vezes, os getters aparecem em um formulário, como "isLobster" em código Java, e o Eclipse usa essa convenção quando você clica com o botão direito do mouse no campo e acessa Fonte > Gerar getters/setters. No entanto, esteja ciente de que o Cloud Endpoints gera "getLobster" no modelo cliente.

Métodos de API

  • O atributo “nome” para anotação @Api deve começar com uma letra minúscula. É preciso formatá-lo como um nome de variável Java, ou seja, letras concatenadas e alfanumérico.
  • Os métodos estáticos não são suportados nas classes Endpoints. Se for necessário ter um método estático, ele deve ser privado.
  • Se você não especificar o atributo httpMethod na assinatura @ApiMethod, o Cloud Endpoints fará um palpite para você, com base no nome do seu método.
  • Use a anotação @ApiNamespace para ter mais controle sobre os nomes dos seus pacotes na biblioteca. Caso contrário, a API usará os nomes de pacotes padrão.
  • Os parâmetros do método na biblioteca cliente gerada estão em ordem alfabética, independentemente da ordem original no método de back-end. Como resultado, tenha cuidado ao editar seus métodos, especialmente se houver diversos parâmetros do mesmo tipo. O compilador não será capaz de detectar erros de ordenação de parâmetro.
  • Listas vazias retornadas chegarão como nulas do lado do cliente. Verifique se há listas assim.
  • Métodos sobrecarregados em uma classe não são suportados.

Sobre os tipos de parâmetros compatíveis

  • A maioria dos tipos de parâmetro deve ser serializável, pois JSON requer objetos serializáveis.
    • Também é possível usar qualquer JavaBean, o que inclui qualquer elemento do seu modelo, uma vez que os objetos do modelo devem estar em conformidade com o padrão JavaBean, conforme descrito acima.
  • Um conjunto limitado de tipos não precisa ser serializável, mas deve ser anotado com @Named. São os tipos: String, int/Integer, boolean/Boolean, long/Long.

Sobre os tipos de retorno compatíveis

  • Os tipos de retorno devem ser serializáveis. Os parâmetros de método compatível @Named, como Strings e Booleans, não são suportados como tipos de retorno. Uma exceção é que você pode retornar void, caso em que o método gerado do lado do cliente retorna uma Void.class.
  • Listas de tipos serializáveis. Na verdade, o que Endpoints geram é uma nova classe JSON "Collection" semelhante a todas as outras classes do modelo cliente. Por exemplo, se você tiver um objeto no modelo representado por Lobster.java e desejar retornar um List<Lobster>, a biblioteca terá um LobsterCollection.java. Para recuperar a lista de Lobsters no cliente, chame getItems().
  • Mapas de tipos serializáveis. Semelhante à maneira como lida com listas, Endpoints gera um "JsonMap" para mapas.
  • Os tipos de subclasse geram o tipo de interface. Por exemplo, caso seu método de back-end retorne um ArrayList, será uma lista no seu cliente. Provavelmente, você está usando a interface para definir suas variáveis e retornar tipos de qualquer maneira, mas fique ciente disso nos casos em que deseje usar a subclasse.

Geração da biblioteca

Quando você começar a gerar a biblioteca de Cloud Endpoints API, algumas características do Plug-in do Google para o Eclipse podem ajudá-lo:

  • Começar do zero é muito fácil. Quando você gera seu primeiro back-end do App Engine do seu projeto Android, o Eclipse ajusta as configurações para conectar os dois. Essas configurações dizem ao Eclipse para colocar o novo código de biblioteca gerado na pasta correta do seu projeto Android. Para mais detalhes, consulte a seção Geração de pontos de extremidade.
  • Se você já tiver um aplicativo do Google App Engine que deseja vincular ao seu front-end do Android, edite as configurações do projeto do Eclipse.
    • No seu projeto do Eclipse para seu cliente Android, há uma pasta oculta chamada ".settings/". O arquivo com.google.gdt.eclipse.appengine.swarm.prefs na pasta de configurações deve conter uma linha: “connectedProject=YourAppEngineBackend”, em que “YourAppEngineBackend” é o nome do seu back-end.
    • No seu projeto do Eclipse para seu back-end do App Engine, há também uma pasta oculta chamada ".settings/". O arquivo com.google.gdt.eclipse.appengine.swarm.prefs na pasta de configurações deve conter uma linha: “connectedProject=YourAndroidClient”, em que “YourAndroidClient” é o nome do seu cliente de front-end.
  • Resolver erros ao gerar a biblioteca do Cloud Endpoints analisando o registro de erros do Eclipse. Quando você começar a desenvolver com o Cloud Endpoints, poderá encontrar problemas ao gerar a biblioteca pela primeira vez. Pode ocorrer o erro "A geração do Cloud Endpoints encontrou erros e não foi concluída." Para ver uma descrição mais completa do erro, selecione Janela > Mostrar visualização > Registro de erros. Dê um duplo clique em cada erro para visualizar uma versão mais detalhada.
  • Certificar-se de que a versão do Eclipse tenha um plug-in do Google compatível com a versão do SDK do App Engine. Se você ainda encontrar erros estranhos, como "método não encontrado", depois de ler o registro de erros, pode ter ocorrido uma incompatibilidade entre as versões. Quando tiver dúvida, atualize para as últimas versões do Plug-in do Google, do SDK do App Engine e do SDK do Android. Se ainda tiver problemas, tente mudar para versões mais recentes do Eclipse.
  • Recriar sua biblioteca periodicamente ou após cada alteração importante. Quando você altera o atributo de nome de @ApiMethod ou qualquer outra coisa que cause mudanças no lado do cliente, deve atualizar a biblioteca. Pense nisso desta maneira: você é o engenheiro do código do servidor e do código do cliente. Às vezes, como engenheiro de back-end, você fará alterações significativas e, como engenheiro de front-end, terá que adaptá-las.
  • Configurar um ambiente de desenvolvimento local, se você quiser testar localmente. Ao gerar a biblioteca de pontos de extremidade, uma classe é criada automaticamente, com o nome CloudEndpointUtils. Se desejar fazer o teste local, defina LOCAL_ANDROID_RUN = true e chame CloudEndpointUtils.updateBuilder(builder) em todo lugar que instanciar um serviço de ponto de extremidade. Dessa forma, você pode alternar facilmente entre versões locais e implantadas, alternando a variável LOCAL_ANDROID_RUN.

Criação do seu modelo para o armazenamento de dados

Lembre-se de tornar seu modelo compatível com o armazenamento de dados se desejar usar o armazenamento permanente de fácil acesso que o armazenamento de dados oferece, que funciona muito bem com o Cloud Endpoints. Encontre aqui uma lista completa de propriedades e tipos do armazenamento de dados. A seguir são descritos alguns dos destaques e ambiguidades do armazenamento de dados em mais detalhes:

  • Há várias opções para interagir com o armazenamento de dados.O JPA / JDO proporciona uma excelente camada de abstração para quando o modelo salvo é simples. A Datastore API de baixo nível oferece a você mais controle sobre como seus objetos persistem. Se você usar a Datastore API de baixo nível, poderá e precisará controlar manualmente as alterações no seu modelo. Isso tem vantagens e desvantagens.
  • Listas vazias persistem como valores nulos antigos simples. Vale ressaltar que, mesmo se você verificar esses valores nulos e anexar uma lista vazia ao objeto que é enviado ao cliente, ainda chegará como nula no cliente.
  • Para persistir grupos de objetos, lembre-se que o armazenamento de dados captura apenas objetos do tipo Coleção ou suas subclasses. Os itens no Coleção devem ser de um dos outros tipos suportados ou devem ser serializáveis em um Blob.
  • Sem matrizes. Como mencionado acima, o armazenamento de dados terá valores do tipo Coleção. Portanto, faça uso deles em vez de matrizes.
  • O armazenamento de dados não distingue os tamanhos dos números, como int32 e int64. Por exemplo, se persistir uma lista de números inteiros, será armazenada como List<Long>.

Você não é obrigado a usar o armazenamento de dados. Um dos benefícios do App Engine é que você pode misturar e combinar vários serviços em nuvem. Serviços como Cloud SQL e JPA, que seria uma ótima opção, estão disponíveis para uso. Como alternativa, você pode explorar o Google BigQuery, se for o tipo de armazenamento de dados que você precisa. O Google Cloud Storage também é uma boa maneira de armazenar vários tipos de dados.

Para desenvolvedores Android iniciantes

Se você for novo no Android, perceberá que o Cloud Endpoints simplifica tudo e permite que você concentre suas energias no próprio aplicativo, destas maneiras, por exemplo:

  • Faça chamadas para a API de forma assíncrona em threads não-IU (interface de usuário). O Android tem um thread principal em execução, muitas vezes denominado "thread IU", e não convém que fique suspenso esperando por respostas.
    • AsyncTasks são seus amigos. Os AsyncTasks que estão vinculados a uma atividade pode ser destruídos quando a atividade também for, por isso cabe a você se certificar de que esse comportamento esteja planejado. Procure por artigos sobre "rotação da tela do AsyncTasks", por exemplo, para encontrar muitas ideias para lidar com isso.
    • Se você precisar usar um objeto que deve ser retornado pela tarefa, use o método de substituição onPostExecute da tarefa. Se você solicitar um objeto Lobster da API usando um AsyncTask e tentar usá-lo imediatamente na próxima linha de código, por exemplo, Lobster lobster = new GetLobsterAsyncTask().execute(lobsterName); lobster.dance();, receberá uma exceção. Assim, não apenas sua lagosta "não dançará", como "jogará um NullPointerException em você”.
    • Se você quiser que a IU espere até o retorno da tarefa, use um ProgressDialog no onPreExecute e métodos onPostExecute.
  • Monitore as atividades e os ciclos de vida dos fragmentos. Se você tiver problemas com isso, substitua um número dos métodos de ciclo de vida importantes e coloque declarações de registro de depuração neles, apenas para ver se e quando suas atividades estão sendo pausadas, reiniciadas, destruídas e assim por diante. Aqui está uma lista completa desses métodos, caso você precise deles.
    • Por exemplo, quando você inicia uma segunda atividade com startActivityForResult, pode ver que a primeira coisa que o ciclo de vida atinge, quando essa segunda atividade termina, é o onActivityResult da primeira atividade, antes de onCreate/Resume.
  • Iniciar uma nova atividade no meio de um método não dispensa o restante da linha de lógica de ser executado. Isso se aplica a startActivity, startActivityForResult ou outros. Mesmo que você não possa vê-lo, o método será concluído, bem como quaisquer outros métodos que ele chame. A atividade que contém o método de execução não desaparece até você destruí-la ou colocá-la para fora da pilha.
  • Às vezes, basta excluir o arquivo R.java. Em outras palavras: apagar e recriar seu projeto. Ocasionalmente, você encontrará alguns problemas em que "não é possível transmitir Lobster em Tree." "Mas não estou tentando", você diz, "porque que isso é bobagem, Lobster e Trees não fazem parte do mesmo reino!" Isso pode significar que o arquivo R.java, que contém ponteiros para valores definidos na pasta res/, ficou um pouco misturado. A recriação deve corrigi-lo.
  • As regras de layout do Android não são iguais às do CSS. Frequentemente, elas podem ser muito semelhantes, por isso você pode se sentir muito à vontade. No entanto, às vezes há pequenas diferenças. Por exemplo, as margens não são recolhidas em Android, portanto, verifique o caso antes de investir um grande esforço no seu layout.
  • Dalvik não é exatamente Java. No entanto, o Dalvik usa a sintaxe Java e o Android é compatível com um grande subconjunto de bibliotecas Java padrão, como java.util e java.io. Não é possível importar qualquer biblioteca Java de terceiros e esperar que funcione bem com Android.
  • Criar uma nova atividade com o assistente "Criar uma nova" do Eclipse gera/modifica cinco arquivos:
    • AndroidManifest.xml: adiciona uma nova entrada <activity>
    • res/values/strings.xml: adiciona uma string de título à atividade
    • res/menu/your_activity.xml: cria esse arquivo
    • res/layout/your_activity.xml: cria esse arquivo
    • YourActivity.java: cria esse arquivo com alguns métodos de inicialização, como a substituição onCreate
  • Use a classe padrão Logger para fazer login no App Engine. É possível visualizar a saída no console do App Engine. O Android usa seu próprio sistema de login e sua saída pode ser vista em uma das visualizações do Eclipse chamada "LogCat".
  • Se você encontrar erros EOFException ocasionais, é provável que isso ocorra devido a conexões keepAlive no Android. Encontre várias soluções no StackOverflow.

Palavra final

Você leu sobre diversos detalhes que pode encontrar, mas a melhor maneira de aprender é colocando em prática. O Cloud Endpoints é extremamente flexível e intuitivo. Ele assume a responsabilidade de desenvolver interações e protocolos do servidor cliente e permite que você se concentre no desenvolvimento da lógica do seu aplicativo. O melhor de tudo, isto não está no caminho: depois que você começa a desenvolver seu aplicativo, fazer o cliente se comunicar com o servidor é tão fácil que você se esquece de que nem sempre foi assim tão simples.

Boa sorte!

Uso do @ApiMethod

Há uma ótima tabela para guiá-lo pelas anotações dos métodos de Java API. Estes são alguns detalhes adicionais aprofundados:

O caminho de URI do REST do seu ponto de extremidade prefere usar estes atributos, nesta ordem:

  1. Seu atributo de caminho.
    • Em path = lobsters: “lobsters” se tornará seu caminho REST.
  2. A segunda parte do seu atributo “nome”.
    • Em name = lobsters.assignName: “assignName” se tornatá seu caminho REST.
  3. O nome do método.
    • Em public void assignName(Lobster lobster): “assignName” se tornará seu caminho REST.

Quanto ao atributo name, pense na primeira parte desse atributo como sua classe de recursos. Pense na segunda parte como o método que você deseja executar. Aqui estão alguns exemplos do atributo name:

  • name = lobsters.assignName no seu back-end será consumido com service.lobsters().assignName(param)no seu código cliente.
  • name = lobsters no seu back-end será consumido com service.lobsters(param)no seu código cliente.

Exemplo 1

Com essas regras em mente, aqui está um exemplo típico de como pode ser um método no seu back-end do App Engine. O recurso lobster é pluralizado:

@ApiMethod(path = “lobsters/assign/[a]name}[a]”, name = "lobsters.assignName",
httpMethod = HttpMethod.POST)
public void makeLobsterANewName(String name) {...}

Em seguida, a biblioteca cliente gera este código para usar no seu cliente:

public Lobsters lobsters() {...}
  public class Lobsters {
  public AssignName assignName(String name) {...}
  public class AssignName extends CloudmazeRequest {
    private static final String REST_PATH = “lobsters/assign/{name}”
    protected AssignName(String name) {...}
  }
}

Para consumir o ponto de extremidade, escreva este código:

String name = “Bob the Lobster”;
service.lobsters().assignName(name).execute();

Exemplo 2

Aqui está um exemplo típico de um método do tipo CRUD mais simples para um recurso Lobster no seu back-end do App Engine. O recursos Lobster é pluralizado e não há nenhum atributo HttpMethod, pois o método "Update" utiliza como padrão HttpMehod.PUT:

@ApiMethod(path = “lobsters”, name = "lobsters.update")
public void updateLobster(Lobster lobster) {...}

A biblioteca cliente gera este código para usar no seu cliente:

public Lobsters lobsters() {...}
  public class Lobsters {
  public Update update(String name) {...}
  public class Update extends CloudmazeRequest {
    private static final String REST_PATH = “lobsters”
    protected Update(String name) {...}
  }
}

Para consumir o ponto de extremidade, escreva este código:

Lobster bobLobster = new Lobster();
service.lobsters().update(bobLobster).execute();
