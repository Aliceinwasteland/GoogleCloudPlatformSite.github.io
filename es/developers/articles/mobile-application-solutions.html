Soluciones para móviles en Google Cloud Platform

Presentación

Más de mil millones de personas utilizan smartphones y tablets en su día a día. Esta situación brinda una gran oportunidad y hace crecer la demanda de soluciones para móviles. Como los usuarios pueden elegir entre cientos de miles de aplicaciones, es importante que los desarrolladores para móviles, para cumplir sus objetivos, creen experiencias de usuario convincentes, atractivas y con conexiones. Este objetivo suele hacer necesario contar con componentes o servicios de backend para transmitir datos pertinentes a la aplicación, para proporcionar notificaciones push, para permitir que los usuarios interaccionen entre ellos, etc.

Con Google Cloud Platform te resultará muy fácil crear un backend para tu solución para móviles. Puede que empieces con algo sencillo con la esperanza de obtener una gran cantidad de usuarios; o quizá ya tengas una gran base de clientes a la que quieras ofrecer una nueva experiencia móvil. Sea cual sea tu caso, puedes desarrollar un backend que se pueda escalar para satisfacer la creciente demanda. Si aprovechas la infraestructura de Google, puedes centrarte en la implementación de los escenarios de la aplicación, sin tener que preocuparte por tareas como la gestión de los equipos o el equilibrio de la carga.

Requisitos

Entre los requisitos habituales de las soluciones para móviles atractivas se incluyen los siguientes:

  • compatibilidad con dispositivos iOS y Android a través de aplicaciones nativas,
  • almacenamiento, obtención y procesamiento de datos fuera de los dispositivos móviles,
  • organización de notificaciones push en dispositivos iOS y Android,
  • conocimiento de la ubicación geográfica y búsqueda por proximidad geográfica,
  • autenticación de usuarios,
  • alta escalabilidad.

Es fácil satisfacer estos requisitos si la solución para móviles se crea en Google Cloud Platform.

Ámbito

En este artículo se presentan los componentes clave de una solución para móviles creada en Google Cloud Platform. Incluye tanto componentes que son comunes a la mayoría de las soluciones para móviles como componentes opcionales que pueden ser útiles en contextos más específicos.

Este artículo está dirigido a arquitectos y a desarrolladores que tengan interés en crear sus propios backends y en diseñar la comunicación entre los clientes móviles y el backend. Más allá de proporcionar información pertinente para la comunicación con el backend, para autenticar usuarios y para recibir notificaciones push, la creación de clientes móviles no se abarca en este documento.

Información general de nivel alto

Estos son los componentes clave de la arquitectura de la solución para móviles propuesta:

  • clientes para móviles: cualquier combinación de clientes Android, iOS y HTML/JavaScript,
  • aplicación de backend móvil, que se encarga de atender las solicitudes de los clientes,
  • capa de comunicación entre los clientes móviles y el backend,
  • varios servicios en la nube para almacenar los datos de la aplicación, para entregar las notificaciones push, etc.

En este documento se te indica cómo seguir los pasos detallados necesarios para crear una solución para móviles, hasta llegar a un diagrama arquitectónico completo que se encuentra al final.

Tu backend móvil[1] en la nube

La capacidad de tener tu propio backend quiere decir que puedes ejecutar el código personalizado en el backend y no estar limitado únicamente a llamar a los servicios de terceros desde las aplicaciones de cliente móvil.

Si sacas provecho de Google Cloud Platform, te resultará muy fácil tener tu propio backend en la nube. Google App Engine es una plataforma perfecta para ejecutar el código, sobre todo el código al que se puede llamar desde los dispositivos de cliente móvil.

Función principal del código personalizado en el backend

El método de creación del backend, tal como se muestra en este documento, te permite centrarte en el desarrollo de los escenarios principales. Tus dos tareas principales son la definición de los recursos específicos del dominio y la implementación de una lógica personalizada necesaria para algunas acciones específicas en estos recursos. Tus clientes móviles podrán acceder fácilmente a estos recursos e invocar estas acciones a través de tu API, que es muy sencilla de mostrar desde tu backend móvil, tal como se describe más adelante.

Tu backend móvil puede obtener algunos de los recursos de los servicios externos sin almacenarlos en la aplicación. Sin embargo, por lo general, la solución para móviles tendrá por lo menos algunos recursos que se administrarán y se conservarán como parte de la solución.

Por ejemplo, si creas una aplicación que incluye reseñas, podrías definir "Reseña" como uno de tus recursos. Si la aplicación permite hacer compras desde móviles, podrías incluir reseñas de los productos en el inventario. Si en la aplicación se enumeran películas, podrías incluir reseñas de películas. Si desarrollas un juego informal, podrías definir PartidaCompletada como una entidad con atributos como la puntuación, la fecha, el ID de usuario, etc.

A continuación, puedes definir las acciones de estos recursos. Por ejemplo, insertar una reseña nueva, añadir una puntuación de utilidad a una reseña, obtener las reseñas según su puntuación de utilidad u obtener las puntuaciones más altas de un día.

En función de tu situación y de tus necesidades, puede que la implementación de estas acciones y del código personalizado sea importante. El código, además de gestionar de forma síncrona las solicitudes de API sencillas por parte de los clientes, también puede servir para muchas más cosas. Podría interesarte hacer procesamientos en segundo plano, permitir las búsquedas basadas en proximidad geográfica o habilitar las notificaciones push. Todo esto es posible si tu backend móvil se ejecuta en Google Cloud Platform. Así, puedes centrarte en escribir código específico de la aplicación y no código de infraestructura.

Liberación de preocupaciones

Si el código de tu backend móvil se ejecuta en Google App Engine, según el diseño recomendado, la infraestructura de Google te ofrece las siguientes ventajas:

  • escalabilidad automática,
  • equilibrio de carga automático entre las instancias del backend móvil,
  • publicación y almacenamiento en la memoria caché de contenido estático,
  • funciones de fiabilidad, rendimiento y seguridad,
  • supervisión y administración,
  • protección DoS configurable,
  • integración con Cuentas de Google para la autenticación OAuth2 (tanto para clientes Android como iOS).

Comunicación entre los clientes móviles y el backend móvil

Hay muchos patrones posibles para la comunicación entre los clientes móviles y el backend. Sin embargo, durante la última década, el modelo basado en REST se ha convertido en el predominante.

Google Cloud Platform ofrece una nueva tecnología de grandes posibilidades llamada Google Cloud Endpoints[2] que simplifica el proceso. Para ello no solo muestra la API REST de tu backend móvil, sino que también utiliza las API REST desde clientes Android, iOS y JavaScript. También proporciona autenticación basada en OAuth2, por lo que el código del backend móvil puede averiguar la identidad del autor de la llamada. El servicio Cloud Endpoints aprovecha la infraestructura de Google para que puedas contar con ella para escalar la aplicación y así satisfacer la demanda. Puesto que te permite utilizar la compatibilidad con herramientas para generar automáticamente código repetitivo necesario, puedes centrarte en escribir código específico para los escenarios de tu aplicación.

Sigue la documentación de Google Cloud Endpoints para obtener información sobre cómo realizar anotaciones en el código de la aplicación para que la API se muestre a través de Google Cloud Endpoints; sobre cómo configurar la autenticación OAuth2; y sobre cómo generar clientes iOS y Android de escritura fiable, así como clientes JavaScript básicos. Si utilizas Eclipse, puedes sacar provecho de la compatibilidad con más herramientas integradas que el complemento de Google para Eclipse te proporciona.

Aunque puedes crear el backend móvil sin utilizar Google Cloud Endpoints (y muchas partes de este documento seguirían siendo pertinentes), la solución propuesta da por supuesto que sí se utiliza.

Creación de la solución para móviles

Conceptos básicos

Como se ha mencionado en secciones anteriores, estos son los componentes esenciales (figura 1) de la arquitectura de la solución para móviles propuesta:

  1. clientes móviles Android o iOS,
  2. Google Cloud Endpoints utilizado para la comunicación entre los clientes y el backend a través de la API REST con autenticación OAuth2 opcional,
  3. el código de tu aplicación de backend móvil ejecutado en Google App Engine y encargado de atender las solicitudes de los clientes.

Almacenamiento de datos

Un requisito habitual de una solución para móviles con un backend es almacenar los datos fuera de los dispositivos cliente.[3] Estos datos se pueden clasificar en dos grupos (figura 2):

  1. Objetos grandes y en general binarios (por ejemplo, imágenes)

    Estos objetos pueden formar parte de la solución; por ejemplo, las imágenes de productos para un inventario dinámico de regalos ofrecidos por tu aplicación para móviles. O si quieres que los usuarios puedan subir sus fotos, sus avatares y otras imágenes. En ambos casos, para almacenar este tipo de datos se puede usar Google Cloud Storage, un servicio que almacena objetos y archivos con tamaños que pueden ascender a terabytes. Consulta también la sección Publicación y procesamiento de imágenes que hay más adelante.

  2. Propiedades y entidades más concretas

    Estas propiedades y entidades pueden incluir información sobre el último nivel completado en un juego para móviles, información sobre los usuarios, el registro de dispositivos del usuario para notificaciones push, registros de compras integradas en la aplicación u otros datos específicos de la aplicación.

    Las propiedades de estas entidades también pueden incluir una referencia (por ejemplo, un nombre de objeto y, opcionalmente, un nombre de segmento o una URL) a objetos almacenados en Google Cloud Storage.

    Con el backend móvil alojado en Google App Engine, el lugar lógico para almacenar este tipo de datos es el almacén de datos de App Engine. Proporciona un almacén de datos de objetos NoSQL[4] sin esquema con un motor de consulta y transacciones atómicas. Estas entidades suelen asignarse a los recursos mostrados a través de la API de Google Cloud Endpoints. Cuando utilices anotaciones adecuadas, puedes aprovechar el complemento de Google para Eclipse a fin de generar código básico para acciones habituales de enumeración, obtención, inserción, actualización o eliminación mediante el almacén de datos de App Engine.

Optimización del acceso a los datos con Memcache

Si tus clientes móviles solicitan a menudo los mismos datos al backend, la mejor solución no es obtener estos datos desde el almacenamiento persistente cada vez que haya una solicitud. Por ejemplo, si creas una aplicación para móviles que muestra reseñas de películas, es probable que la mayoría de los usuarios quiera obtener reseñas sobre una de entre unas pocas películas que se encuentren en cartelera en ese momento y habrá muchas menos solicitudes para los miles de reseñas de películas restantes que el backend tenga almacenadas en un almacenamiento persistente.

Un backend que se ejecute en Google App Engine puede utilizar un servicio de almacenamiento en memoria caché de datos distribuido llamado Memcache para almacenar y obtener datos a los que se acceda a menudo (figura 3). Si utilizas Memcache, no solo permites que tu solución para móviles tenga un mejor rendimiento y una mejor escalabilidad, sino que también se reduce el coste de acceder al almacén de datos de App Engine, ya que la aplicación enviará menos solicitudes al almacén de datos de App Engine.

Procesamiento asíncrono con colas de tareas

A veces no se puede, ni conviene, hacer todo el trabajo asociado a una solicitud enviada desde un cliente móvil antes de que se tenga que enviar la respuesta al dispositivo del usuario. Pongamos por caso que estés desarrollando una aplicación social que envíe una notificación a los amigos del usuario cuando el usuario registre visita en un restaurante. Es probable que quieras registrar la visita de forma síncrona, a la vez que registras el hecho de que la notificación se tiene que procesar, y que después quieras enviar de inmediato la respuesta de confirmación al cliente sin tener que esperar a que se procesen todas las notificaciones.

El código que se ejecuta en Google App Engine puede utilizar un servicio llamado colas de tareas para poner en cola información sobre el trabajo que hay que hacer de forma asíncrona (es decir, las "tareas"). En el ejemplo anterior, tu aplicación podría poner en cola una tarea para notificar a los amigos de David que David ha registrado visita en el restaurante Gran Muralla a las 20:35.

Normalmente, tendrías que configurar el servicio de cola de tareas para que enviara las tareas en cola (modelo push) a tu código que está en ejecución en Google App Engine como solicitudes HTTP a una URL seleccionada. Tu aplicación puede procesar estas solicitudes a través de controladores de solicitudes web normales. En el ejemplo anterior, el controlador de solicitudes fue capaz de obtener la información de la tarea de la solicitud HTTP, de buscar los amigos de David y de luego notificar a cada uno de ellos mediante el envío de una notificación push. Puedes configurar que las tareas se procesen con las mismas instancias de Google App Engine que están procesando las solicitudes de clientes móviles o que se procesen con instancias de dedicación exclusiva.

De forma alternativa, el código puede retirar las tareas de las colas. Para retirarlas, tienes que ejecutar el código del backend móvil en Google App Engine o tienes que ejecutar el código fuera de App Engine, que se puede usar para procesar tareas con instancias de Google Compute Engine.

En cualquier caso, el procesamiento de tareas suele ser la segunda responsabilidad más habitual que recae sobre el código personalizado que se ejecuta en Google App Engine, además de procesar las solicitudes de los clientes móviles a través de Google Cloud Endpoints, tal como se ha mencionado en secciones anteriores.

Notificación push

La notificación push es muy importante para la experiencia de usuario general en los dispositivos móviles. En función de la plataforma cliente, tu solución para móviles puede usar esta tecnología para mostrar mensajes automáticos y otras notificaciones en los dispositivos de los usuarios aunque la aplicación para móviles no haya estado en ejecución. También se puede utilizar para seguir transmitiendo los datos pertinentes a la aplicación cliente móvil.

Con el código del backend móvil en ejecución en Google App Engine, puedes organizar el envío de notificaciones push a los usuarios gracias a las siguientes características tecnológicas que se te brindan:

  • Mensajería en la nube de Google para Android: un servicio de Google que te permite enviar datos a los usuarios que tengan instalada tu aplicación cliente en sus dispositivos con tecnología Android. Si utilizas Eclipse, puedes hacer que el complemento de Google para Eclipse genere código básico.
  • API de sockets[5] para enviar notificaciones push con el servicio de notificaciones push de Apple® a los usuarios de iOS.

Un ejemplo de esto podría ser el envío de notificaciones push a los usuarios de tu solución para móviles cuando un amigo registre visita en un restaurante o cuando publique una reseña (figura 5).

Publicación de contenido estático

En algunos casos, puede que también puedas sacar provecho de la capacidad de empaquetar contenido estático como parte de la aplicación de Google App Engine para que luego lo publique la infraestructura de Google App Engine mediante servidores y cachés de dedicación exclusiva sin que el código de tu backend móvil tenga que participar. Esta situación se suele dar más en clientes HTML o JavaScript que en aplicaciones iOS o Android nativas, y se puede aplicar a recursos tales como imágenes, hojas de estilo CSS o código JavaScript del navegador.

Publicación y procesamiento de imágenes

Si tu solución para móviles incluye un número de imágenes estáticas que se pueda gestionar, puedes empaquetarlas (a menudo en varias resoluciones) como parte de tus aplicaciones Android y iOS nativas. Esto concierne, por ejemplo, a varios iconos que formen parte de tu aplicación y que se utilicen para crear la interfaz de usuario de las aplicaciones cliente.

Otra categoría de imágenes son aquellas que no forman parte de tus aplicaciones, sino que más bien están relacionadas con los datos que tu aplicación procesa, tales como miniaturas de películas en tu aplicación relacionada con películas, imágenes de productos que se pueden regalar a través de la aplicación o avatares en las aplicaciones sociales o de juegos. Si puedes almacenar este tipo de imágenes en resoluciones y en formatos optimizados para tus clientes móviles, puedes subirlas a Google Cloud Storage y después proporcionar a los clientes móviles URL directas a estas imágenes. Este método permite una fácil escalabilidad, ya los clientes móviles pueden recuperar estas imágenes directamente desde el almacenamiento en la nube escalable y sin que tu código del backend móvil tenga que participar.

En algunos casos, puede ser buena idea manipular y procesar las imágenes además de almacenarlas. Por ejemplo, puede que quieras publicar las imágenes en tamaños ajustados de manera dinámica en función de la resolución de los dispositivos cliente. En tales situaciones, puedes utilizar el servicio de imágenes de Google App Engine para modificar el tamaño de imágenes, para girarlas, para darles la vuelta y para recortarlas, así como para mejorar las fotografías con un algoritmo predefinido. Estas acciones se pueden realizar en imágenes de Google Cloud Storage, entre otras fuentes. A continuación, las imágenes modificadas se pueden publicar para los clientes directamente desde la infraestructura de publicación de imágenes dinámica que Google proporciona sin que el código del backend móvil se vea implicado. Tu código solo se encarga del uso de la API del servicio de imágenes para obtener la URL de imagen correcta y para proporcionar esta URL a los clientes móviles.

Búsqueda de texto completo y por proximidad geográfica (basada en la ubicación)

Si quieres que tus usuarios puedan realizar búsquedas de texto completo a través de los datos estructurados que tu solución para móviles administra, puedes sacar provecho de la API de búsqueda[6] de Google App Engine. Por ejemplo, si la aplicación proporciona acceso a un inventario grande y dinámico de elementos, los usuarios pueden sacar provecho de la posibilidad de buscar elementos mediante la escritura de palabras que forman parte del nombre o de la descripción de un elemento, incluso si proporcionas una práctica interfaz de usuario táctil con categorización, paginación, etc.

Para utilizar la API de búsqueda, puedes construir "documentos" que contengan datos disponibles para búsquedas y luego añadir estos "documentos" a un índice. A continuación, tu backend móvil puede construir consultas de búsqueda, incluidas las consultas de búsqueda de texto completo, y ejecutarlas tomando los "documentos" del índice como referencia. En el ejemplo anterior, cada elemento del inventario se podría asignar a un "documento" del índice con propiedades como el nombre, la descripción, etc.

Tu solución para móviles también puede realizar una búsqueda por proximidad geográfica; por ejemplo, si quieres obtener una lista de tiendas cercanas. Si quieres evitar la complejidad que supone crear funciones de este tipo personalmente, solo tienes que aprovechar la API de búsqueda para la búsqueda por proximidad geográfica.[7] El método básico es muy similar al proceso descrito anteriormente para la búsqueda de texto completo. Solo tienes que añadir la ubicación geográfica como una de las propiedades del documento y luego incluir una condición de proximidad geográfica en las consultas de búsqueda.

Mantenimiento personalizado y otras tareas programadas

Además de procesar solicitudes de los clientes móviles de forma síncrona a través de Google Cloud Endpoints y de forma asíncrona mediante colas de tareas, puede que el código de tu backend móvil tenga que hacer algún tipo de procesamiento en varios intervalos periódicos o semiperiódicos. Por ejemplo, si quieres que la solución para móviles envíe notificaciones push a los usuarios móviles cada mañana con una oferta local pertinente o si quieres que tu solución lleve a cabo varias tareas de mantenimiento específicas de la aplicación cada día o varias veces al día. Por ejemplo, la aplicación podría ofrecer suscripciones de nivel superior. Cuando las suscripciones caduquen o estén a punto de caducar, podrías llevar a cabo varios procesos relacionados como, por ejemplo, enviar un correo electrónico mediante la API de correo, desactivar las cuentas de usuario, etc. Puede que también quieras que el backend móvil actualice algunos datos almacenados en memoria caché en Memcache cada diez minutos o que actualice las estadísticas de los mejores jugadores cada hora.

Si el código de tu backend móvil se ejecuta en Google App Engine, la configuración de este tipo de tareas programadas es sencilla con el servicio cron. Puedes configurarlo para invocar algunas URL en tu backend móvil en los intervalos solicitados o a horas específicas.

Almacenamiento y análisis de registros de la aplicación

Los registros de la aplicación pueden proporcionar gran cantidad de información valiosa acerca de la forma en la que los usuarios utilizan tu solución para móviles y acerca del comportamiento de tu backend móvil. Si el código de tu backend móvil se ejecuta en Google App Engine, puedes usar la API de registros para obtener los registros de la aplicación. A continuación, puedes subirlos a Google Cloud Storage y transferirlos a Google BigQuery. Si tu aplicación cliente móvil permite a los usuarios realizar numerosas interacciones dentro de las aplicaciones sin enviar solicitudes periódicas al backend, puede ser recomendable aumentar los registros de backend con registros que la aplicación cliente pueda recopilar y enviar periódicamente al backend móvil a través de tu API mostrada a través de Google Cloud Endpoints.

Ejecución de código fuera de la zona de pruebas de App Engine

Si creas todo el backend móvil con tiempos de ejecución compatibles con Google App Engine,[8] evitarás tener que hacer frente a problemas de infraestructura como el escalado automático, el equilibrio de carga, la tolerancia a fallos, etc. Sin embargo, puede que tengas requisitos que no sean adecuados para la ejecución de código dentro de la zona de pruebas de App Engine. Por ejemplo, puede que quieras utilizar algunas bibliotecas C/C++, algunos archivos binarios personalizados o algunos programas PERL como parte del procesamiento asíncrono de solicitudes o del procesamiento por lotes.

En estos casos, puedes usar Google Compute Engine[9] para ejecutar las partes de tu backend móvil que no se puedan alojar en Google App Engine.

Cuando se utilice código ejecutado en Compute Engine para el procesamiento por lotes o para el procesamiento asíncrono de solicitudes,[10] el código que se ejecute en Google App Engine puede poner las tareas en una cola para tareas extraídas (o en varias colas) y el código que se ejecute en Google Compute Engine puede extraer estas tareas con la API REST de cola de tareas.[11]

Otros servicios y otras API pertinentes

Google ofrece muchos otros servicios y APIs que se pueden utilizar en las soluciones para móviles. Entre ellos: AdMob, Analytics y varias API de Google Maps, como API de Places, SDK para Android Maps y SDK para iOS Maps. El uso de estos servicios y de estas API queda fuera del alcance de este documento.

Conclusión

Al aprovechar Google Cloud Platform, puedes crear fácilmente un backend para tu solución para móviles. Olvídate de la parte técnica y de la infraestructura. Tu solución puede ser altamente escalable y cumplir los requisitos para crear aplicaciones atractivas. Entre otras cosas, puedes utilizar autenticación de usuario integrada, compatibilidad con aplicaciones nativas Android y iOS, y almacenamiento, recuperación y procesamiento de datos fuera de los dispositivos móviles. Además, puedes organizar notificaciones push y utilizar la búsqueda por proximidad geográfica.

Aplicación de ejemplo

Puedes descargar la aplicación de ejemplo Mobile Shopping Assistant (backend de App Engine, cliente Android y cliente iOS) y leer el tutorial correspondiente para ver cómo se han aplicado las piezas clave de la arquitectura (presentada en este documento) en una situación concreta. La aplicación de ejemplo incluye la API REST mostrada a través de Google Cloud Endpoints con la autenticación de clientes integrada y basada en OAuth, búsqueda por proximidad geográfica, notificación push, procesamiento asíncrono de solicitudes, tareas programadas y publicación de imágenes desde Google Cloud Storage.


Notas

[1] En términos generales, en este documento, "backend" significa la parte de una solución para móviles que se ejecuta fuera de los dispositivos cliente. No se debe confundir con las instancias de backend de Google App Engine, que representan un tipo de instancias que tienen características diferentes a las instancias predeterminadas de App Engine. En muchas soluciones para móviles, las instancias predeterminadas de Google App Engine deberían bastar. Sin embargo, si tienes unos requisitos específicos, puedes crear una solución para móviles con un "backend" formado por cualquier combinación de instancias predeterminadas de App Engine y de instancias de backend.

[2] Desde febrero de 2013, Google Cloud Endpoints es una función experimental.

[3] En el apartado 4.2 de las Condiciones de servicio de Google App Engine se describe tu obligación de proteger la privacidad y los derechos legales de tus Usuarios finales, inclusive la necesidad de proporcionar una política de privacidad que se adecue a la ley en lo referente al uso de los datos que recopiles y de obtener todos los consentimientos necesarios por parte de los usuarios finales para la recopilación, para el uso, para la supervisión o para la revelación de dichos datos.

[4] Si prefieres utilizar una base de datos SQL relacional, puedes usar Google Cloud SQL en vez del almacén de datos o de forma complementaria.

[5] Desde principios de enero de 2013, la API de sockets es una función disponible únicamente para Testers de confianza y no es compatible con la producción.

[6] Desde principios de enero de 2013, la API de búsqueda de App Engine es una función experimental con cuotas gratuitas limitadas.

[7] Desde diciembre de 2012, la búsqueda por proximidad geográfica funciona en las aplicaciones implementadas en App Engine. Sin embargo, no funciona en el servidor de desarrollo local.

[8] Desde diciembre de 2012, App Engine es compatible con Java 5 y 6 y con Python 2.7.3 y 2.5.2, así como con el tiempo de ejecución Go (este último como una función experimental).

[9] Desde diciembre de 2012, Google Compute Engine está disponible para algunos usuarios seleccionados como vista previa restringida.

[10] Google Compute Engine también se puede utilizar en otros escenarios de soluciones para móviles, pero este uso queda fuera del alcance de este documento.

[11] Desde diciembre de 2012, la API REST de cola de tareas es una función experimental.
