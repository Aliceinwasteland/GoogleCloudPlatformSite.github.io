Google Cloud Endpoints para Android

Consejos y prácticas recomendadas de un desarrollador Java

Presentación

Imagina que quieres desarrollar una aplicación para móviles de primera categoría. Echas mano de Google Cloud Endpoints porque has oído que te ahorrará mucho tiempo y mucha codificación. Es una característica bastante nueva y has leído parte de la documentación, de los artículos y de los tutoriales de gran calidad que hay publicados.

Aunque te has armado de esta información, no estás del todo seguro de algunos de los aspectos más sutiles. Cuando intentas generar tu biblioteca de Cloud Endpoints, no entiendes qué parte del código provoca error. ¿Cómo puedes anotar tus métodos de manera que no haya conflictos entre rutas REST? Seguro que codificar las asignaciones con colores entre el código generado y las anotaciones ayudaría.

En este artículo encontrarás respuesta a estas y a otras preguntas. Si no estás atascado pero de todos modos te gustaría conocer algunos consejos, este artículo también es un buen punto de partida en tu caso. Si eres un principiante sin conocimientos previos y puede que aún tengas preguntas del tipo "¿Qué es Cloud Endpoints?", vuelve a leer la información general y las secciones sobre Python o sobre Java. Estas secciones son muy breves y será suficiente con que hagas una lectura rápida hasta que más adelante, cuando estés desarrollando la aplicación, necesites volver a consultarlas.

En las siguientes secciones se analizan algunos consejos generales, detalles prácticos, problemas con los que te puedas encontrar y soluciones para ellos. Se da por supuesto que tienes ciertos conocimientos de Cloud Endpoints, porque este artículo no es una "guía" completa, un "tutorial" ni siquiera un "manual". Considéralo un apéndice muy útil de la documentación principal para desarrolladores. Si ni siquiera estás seguro de por dónde empezar, vuelve a leer la información general en esta página.

Cloud Endpoints en pocas palabras

Cloud Endpoints es un servicio para crear y mostrar API personalizadas con las que tus aplicaciones cliente se pueden relacionar.

Existe para facilitarte la vida como desarrollador. Los ingenieros de Google se han esforzado sobremanera en escribir y en optimizar este servicio para ti, para liberarte de la preocupación que supone escribir uno mismo la mayor parte de los protocolos y de las interacciones cliente-servidor.

Una de las ventajas de ser capaz de crear fácilmente tu propia API es que puedes diseñarla como quieras. Cloud Endpoints te da esta flexibilidad, permite crear API REST o de estilo RPC, o una combinación de ambas. Mejor aún, puedes cambiar el diseño todas las veces que quieras y aplicar esos cambios de manera rápida. Esto es ideal para lograr un estilo de desarrollo ágil.

Unos conceptos sobre REST básico

En REST, la idea básica es mostrar el estado del sistema como una colección de recursos direccionables concretos. Para realizar operaciones, manipulas estos recursos. Un recurso es una entidad que tiene un estado, así como un nombre o un ID. Los recursos se almacenan en tu backend, ya sea Google Cloud Datastore u otro.

Como regla general, con REST el cliente ve representaciones de los recursos a través de solicitudes realizadas al servidor. Para actualizar los recursos, se manipulan las representaciones y se envían las representaciones actualizadas al servidor a través de solicitudes posteriores. Los recursos se pueden representar de una serie de maneras: XML, JSON, CSV o más. De momento, Cloud Endpoints solo es compatible con JSON, que tiende a ser menos detallado y menos vulnerable a los errores humanos que XML.

Es posible que hayas oído hablar de CRUD (siglas en inglés de crear, recuperar, actualizar, eliminar) o de variantes de la misma. CRUD se encuentra en la raíz de REST. Cuando utilizas el Complemento de Google para Eclipse a fin de generar una clase de extremo, estos tipos de métodos básicos se generan en tu nombre en la clase de extremo.

Ni que decir cabe que estas acciones básicas no te limitan. La capacidad de añadir métodos personalizados te proporciona mucha más flexibilidad y te permite realizar operaciones complejas. Por ejemplo, checkForVictory podría realizar varias consultas, además de escribir, en una base de datos.

Configuración: un repaso rápido

En esta sección se ofrece un repaso rápido sobre cómo generar el proyecto y crear extremos en Eclipse. Para obtener más información, consulta la documentación para desarrolladores.

  1. Asegúrate de que los elementos siguientes se hayan descargado y configurado en el equipo:
    1. Eclipse
    2. Complemento de Google para tu versión de Eclipse
    3. SDK de App Engine
    4. SDK de Android

    En el momento en el que se escribió este artículo, las versiones más recientes eran Eclipse 4.2, SDK 1.8.0 de App Engine y SDK r21.01.1 de Android.

  2. Si aún no dispones de una aplicación cliente de Android, crea una en Eclipse.
  3. Crea tu backend de App Engine: haz clic con el botón derecho en tu aplicación Android y selecciona Google > Create App Engine Backend (Google > Crear backend de App Engine).
  4. Puedes generar clases básicas de extremos. Solo tienes que hacer clic con el botón derecho en la clase Java del objeto de datos y seleccionar Google > Generate Cloud Endpoint Class (Google > Generar clase de Cloud Endpoints). Estas clases se pueden seguir modificando en cualquier momento.
  5. Haz clic en la aplicación del backend de App Engine y selecciona Google > Generate Cloud Endpoint Library (Google > Generar biblioteca de Cloud Endpoints) para generar la biblioteca.

Si eres usuario de IntelliJ, tienes Android Studio en IntelliJ como opción alternativa para un entorno de desarrollo de Android. Aunque en este artículo no se analiza esa opción, puedes obtener más información al respecto en el sitio para desarrolladores de Android. Como de momento esta opción se encuentra en versión de vista previa de acceso temprano, ten en cuenta que de vez en cuando pueden aparecer bugs o cambios. Sin embargo, como es sumamente versátil y tiene muy buenas características existentes y previstas, es mejor que no le quites ojo, sobre todo si eres un usuario ávido de IntelliJ.

Fragmentos de código de extremos

Definición de extremos en el backend

El código de extremos de la aplicación de App Engine se puede agrupar en dos categorías: modelo y API.

El primer tipo es tu modelo. Estas clases representan los objetos que hay en el mundo de tu aplicación. Dicho de otro modo, representan los recursos que la API manipulará. Por ejemplo, si el mundo de tu aplicación contiene langostas (lobster) con bigotes (whiskers), puede que tengas un objeto de datos como este:

public class Lobster {
  private int whiskerLength;
  public int getWhiskerLength() {
    return whiskerLength;
  }
  public void setWhiskerLength(int whiskerLength) {
    this.whiskerLength = whiskerLength;
  }
}

El segundo tipo es el código de la API. La parte sobre la API de tu código proporciona los métodos a los que se puede acceder en una aplicación cliente. Estos métodos manipulan los objetos de tu modelo. Hay dos formas de empezar a escribir código para API:

  • Puedes hacer clic con el botón derecho en Lobster.java y seleccionar Google > Generate Endpoint Class (Google > Generar clase de extremo), con lo cual Eclipse generará en tu nombre una nueva clase llamada LobsterEndpoint.java. Esta clase se rellena de anotaciones y de código pregenerados, listos para que se les llame desde el cliente. Estos métodos generados insertan, actualizan y eliminan los objetos Lobster de tu almacén de datos. Para seguir modificando esta clase, añade o elimina métodos.
  • Puedes crear tu propia clase Java y ponerle el nombre que quieras. Siempre y cuando esté bien anotada, Cloud Endpoints la reconocerá como parte de tu API. Ten en cuenta que los métodos privados de instancia no pasarán a formar parte de tu API.

A continuación te presentamos el aspecto que puede tener un ejemplo de clase de extremo:

@Api(name = "lobster")
public class LobsterEndpoint {
  @ApiMethod(name = "lobster.growWhisker", httpMethod = HttpMethod.POST)
  public void growWhisker(Lobster lobster) {
    // grows the lobster’s whisker
    // calculates growth, updates the Lobster in Datastore
  }
  private int calculateWhiskerGrowth(Lobster lobster) {
    // this method is a private utility method
    // it will NOT be visible in your API
  }
}

En general, te recomendamos que no pases mucho tiempo con el código del extremo generado. Es útil y contiene muy buenos patrones para utilizar como punto de partida, pero no tengas miedo de modificarlo para tu proyecto específico. El código es tan maleable como tú quieras. Por ejemplo, las entidades DeviceInfo generadas automáticamente utilizan ID de registro como claves. Aún así, puedes añadir nuevos campos a una entidad DeviceInfo o, si no, utilizar otro campo para la clave.

Uso de extremos desde el cliente

Cuando ya tengas generada tu biblioteca de Cloud Endpoints, el nuevo código que aparece en tu cliente (por lo general en alguna carpeta que indique endpoint-libs en la ruta de archivo) corresponde a las funciones que has proporcionado en el código del backend.

La diferencia en el modelo es que las nuevas clases amplían la clase GenericJson, de modo que se pueden enviar como JSON. Aquí tienes el aspecto aproximado que tendrá la clase Lobster recién generada:

public final class Lobster extends GenericJson {
  private int whiskerLength;
  public int getWhiskerLength() {
    return whiskerLength;
  }
  public Lobster setWhiskerLength(int whiskerLength) {
    this.whiskerLength = whiskerLength;
    return this;
  }
}

El código del cliente de API es una serie de métodos y de clases de solicitud JSON, que hacen que te resulte muy fácil utilizarlo en un cliente de Android. Este puede ser el aspecto de la clase LobsterEndpoint.java:

public Lobster lobster() {...}
public class Lobster {
  public GrowWhisker growWhisker(Lobster lobster) {...}
  public class GrowWhisker extends LobsterRequest {
    private static final String REST_PATH = “growWhisker”
    protected GrowWhisker(Lobster lobster) {...}
    ...
  }
}

A continuación, puedes usar este extremo mostrado con el siguiente código:

service.lobster().growWhisker(lobster).execute();

Si estás familiarizado con las solicitudes HTTP, sabrás que, por cada ruta REST determinada, solo se puede proporcionar una definición de POST, una de GET, etc. Por supuesto que puedes crear varias instancias de POST en la misma API de extremo, pero asegúrate de que sus rutas REST sean diferentes o, de lo contrario, te encontrarás con conflictos. Para obtener más información sobre las anotaciones @ApiMethod, consulta la sección correspondiente del apéndice.

Lo ideal sería que, una vez generada la biblioteca en la aplicación cliente, dejes tranquilo el código de cliente generado. Cualquier cambio que quieras hacer en el modelo o en la API lo tienes que hacer en el backend.

Diseño de una API propia

Modelo

  • Las clases que hay en tu modelo deben tener ciertas características.
    Si quieres utilizar una clase en tu API, estas son las características necesarias:
    • Que se pueda serializar.
    • Que tenga un creador sin argumentos.
    • Que implemente acciones get y set para sus campos.
  • Las clases que hay en tu modelo también pueden contener otros métodos de instancia.
    Ten en cuenta que los métodos que no son acciones get o set nunca verán la luz en el cliente.
  • ¿Cómo puede ser que una definición de clase de objetos incluida en el modelo se marque para que se incluya en el modelo de cliente generado?
    Imagina que tienes una clase Clam.java y algunos campos y acciones get y set (get/setPearl, get/setColor, por ejemplo), pero cuando generas la biblioteca de extremos, Clam.java no aparece en el código generado. ¿Qué le ha pasado? La forma en la que utilizas el objeto Clam en el resto del modelo y en la API del backend determina este comportamiento.
    • Si utilizas la entidad en tu API, forzarás a Cloud Endpoints a reconocer que Clams existen como objetos. 
      @ApiMethod(name = "clam.producePearl", httpMethod = HttpMethod.POST)
public void producePearl(Clam clam) {
  // adds a pearl to the clam
}
    • Si utilizas Clams correctamente en tu modelo, fuerzas a Cloud Endpoints a reconocerlos y a generar Clam.java en tu nombre en el cliente. Para hacer que Cloud Endpoints reconozca Clams como objetos, escribe un método de acción set setClam (Clam clam), que puede ser público, protegido o privado.
      • Los métodos de la acción set deben tener las siguientes propiedades:
        • El nombre del método debe empezar por "set".
        • El objeto que quieras definir se debe encontrar en los parámetros.
          Por ejemplo:

          Salta a la vista que setClam(Tree tree) está definiendo Tree, no Clam, a pesar de que el nombre del método afirme lo contrario. Este método generará una clase Tree.java en el cliente, así como los métodos getTree() y setTree(Tree). No generará un método setClam(Clam).

Acerca de los datos compatibles

  • Enums. Estas funciones del backend se generan como representaciones de String en el código cliente. Dicho de otro modo, si utilizas las funciones enum en el backend y en el cliente, tendrás que definir enum en ambos lugares.
  • Boolean: las acciones get pueden tener la forma "isLobster" en código Java, y Eclipse utiliza esa convención cuando haces clic con el botón derecho en el campo y vas a "Source > Generate Getters/Setters" (Origen > Generar acciones get/set). Sin embargo, ten en cuenta que Cloud Endpoints generará "getLobster" en el modelo cliente.

Métodos de la API

  • El atributo "name" de la anotación @Api debe empezar por minúscula. Debe tener el mismo formato que un nombre de variable Java, es decir, con mayúsculas intercaladas y alfanumérico.
  • Los métodos estáticos no son compatibles con las clases de extremos. Si debes tener un método estático, tienes que definirlo como privado.
  • Si no especificas el atributo httpMethod en la firma @ApiMethod, Cloud Endpoints hará una conjetura de cortesía basándose en el nombre de tu método.
  • Utiliza la anotación @ApiNamespace para lograr un mayor control de los nombres de paquete de tu biblioteca. De lo contrario, tu API utilizará los nombres de paquete predeterminados.
  • Los parámetros del método que hay en la biblioteca cliente generada están en orden alfabético, fuera cual fuera el orden original en el método del backend. Por lo tanto, debes andar con cuidado cuando edites los métodos, sobre todo si hay varios parámetros del mismo tipo. El compilador no podrá detectar errores en el orden de los parámetros.
  • Las listas vacías que se ofrecen llegarán como nulas en el cliente. No te olvides de revisarlas.
  • Los métodos sobrecargados en una clase no son compatibles.

Información sobre tipos de parámetro compatibles

  • La mayoría de tipos de parámetro tienen que permitir la serialización porque JSON exige que los objetos se puedan serializar.
    • También puedes utilizar cualquier clase JavaBean, que lo incluye todo de tu modelo, ya que los objetos de tu modelo se tienen que ajustar al estándar JavaBean según se ha descrito antes.
  • Un conjunto limitado de tipos no necesita permitir la serialización, pero debe estar anotado con @Named. Se trata de los tipos: String, int/Integer, boolean/Boolean, long/Long.

Información sobre los tipos de devolución compatibles

  • Los tipos de devolución tienen que permitir la serialización. Los parámetros del método @Named compatibles (como los parámetros de tipo String y Boolean) no son compatibles como tipos de devolución. Una excepción es que se puede devolver una clase void (vacía), en cuyo caso el método generado en el cliente devolverá Void.class.
  • Listas de tipos que permiten la serialización. En realidad, lo que los extremos generan es una nueva clase JSON "Collection" (colección) que se parece a cualquier otra clase del modelo cliente. Por ejemplo, si tienes un objeto en tu modelo que esté representado por Lobster.java y quieres devolver una lista de Lobster List<Lobster>, la biblioteca tendrá una clase LobsterCollection.java. Para obtener la lista de Lobster en tu cliente, llama a getItems().
  • Mapas de tipos que permiten la serialización. De manera parecida a cómo gestiona las listas, los extremos generan un "JsonMap" para Mapas.
  • Los tipos de subclase generarán el tipo de interfaz. Por ejemplo, si el método de tu backend devuelve ArrayList, será una lista en tu cliente. Huelga decir que, de todos modos, es probable que debieras utilizar la interfaz para definir las variables y los tipos de devoluciones, pero debes tenerlo en cuenta en los casos en los que quieras utilizar la subclase.

Generación de una biblioteca propia

Cuando empieces a generar la biblioteca de API de Cloud Endpoints, aquí tienes algunas características del Complemento de Google para Eclipse que te pueden ayudar:

  • Es muy fácil empezar de cero. Cuando generas tu backend de App Engine por primera vez desde tu proyecto de Android, Eclipse establece las configuraciones para enlazarlos. Estas configuraciones ordenan a Eclipse que coloque el código de la nueva biblioteca generada en la carpeta correcta de tu proyecto de Android. Para obtener más detalles, consulta las secciones sobre generación de extremos.
  • Si ya tienes una aplicación de App Engine que quieras enlazar con tu frontend de Android, tendrás que editar la configuración de tu proyecto de Eclipse.
    • En el proyecto de Eclipse para tu cliente de Android hay una carpeta oculta llamada ".settings/". El archivo com.google.gdt.eclipse.appengine.swarm.prefs de la carpeta de configuración debería contener la línea: "connectedProject=TuBackendDeAppEngine", en la que "TuBackendDeAppEngine" es el nombre de tu backend.
    • En el proyecto de Eclipse para tu backend de App Engine también hay una carpeta oculta llamada ".settings/". El archivo com.google.gdt.eclipse.appengine.swarm.prefs de la carpeta de configuración debería contener la línea: "connectedProject=TuClienteDeAndroid", en la que "TuClienteDeAndroid" es el nombre de tu cliente de frontend.
  • Permite resolver los errores producidos al generar la biblioteca de Cloud Endpoints gracias a la información incluida en el registro de errores de Eclipse. Cuando empieces a desarrollar con Cloud Endpoints, puede que te encuentres con problemas al generar la biblioteca en un primer momento. Es posible que se produzca un error durante la generación de Cloud Endpoints y que no se pueda completar. Para ver una descripción mucho más completa del error, selecciona Window > Show View > Error Log (Ventana > Mostrar la vista > Registro de errores). Haz doble clic en cada error para ver el resultado detallado.
  • Te permite asegurarte de que la versión de Eclipse tenga un complemento de Google compatible con la versión del SDK de App Engine. Si, después de leer el registro de errores, aún se producen errores extraños como que no se encuentra el método, puede que se deba a la incompatibilidad entre versiones. En caso de duda, actualiza a las versiones más recientes del complemento de Google, del SDK de App Engine y del SDK de Android. Si aún hay problemas, prueba a cambiar a versiones más recientes de Eclipse.
  • Puedes volver a generar la biblioteca de manera periódica o después de cada cambio importante. Cuando cambies el atributo de nombre de @ApiMethod, o cualquier otra cosa que provoque cambios en el cliente, tienes que actualizar la biblioteca. Puedes planteártelo como que eres el ingeniero tanto para el código de servidor como para el código de cliente. A veces, como ingeniero del backend, harás cambios importantes y el ingeniero del frontend tendrá que adaptarse.
  • Puedes configurar un entorno de desarrollo local si quieres hacer pruebas locales. Al generar la biblioteca de extremos, se crea una clase automáticamente llamada CloudEndpointUtils. Si quieres hacer pruebas locales, acuérdate de definir LOCAL_ANDROID_RUN = true y llama a CloudEndpointUtils.updateBuilder(builder) en cada lugar que cree instancias de un servicio de extremo. De esta forma, puedes cambiar fácilmente de las versiones locales a las implementadas con solo cambiar la variable LOCAL_ANDROID_RUN.

Diseño de un modelo propio para el almacén de datos

Si quieres utilizar el almacenamiento persistente de fácil acceso que ofrece el almacén de datos (y que funciona tan bien con Cloud Endpoints), no olvides hacer que tu modelo cumpla los requisitos del almacén de datos. Echa un vistazo a esta página, en la que puedes consultar la lista completa de propiedades y tipos de almacén de datos. A continuación se describen más a fondo algunos de los aspectos más destacados y algunas de las ambigüedades del almacén de datos:

  • Tienes varias opciones para relacionarte con el almacén de datos. JPA/JDO proporciona una excelente capa de abstracción en caso de que tu modelo guardado sea sencillo. La API de almacén de datos de bajo nivel te permite controlar mejor la forma de persistencia de los objetos. Si utilizas la API de almacén de datos de bajo nivel, puedes controlar manualmente los cambios de tu modelo y así tendrás que controlarlos. Esto tiene sus ventajas y sus desventajas.
  • Las listas vacías persisten como nulos antiguos sin formato. Vale la pena señalar que, aunque busques estos nulos y adjuntes una lista vacía al objeto que se envía al cliente, seguirá llegando como nulo en el cliente.
  • En el caso de los grupos de objetos persistentes, recuerda que el almacén de datos solo incluye objetos de tipo Collection o de sus subclases. Los elementos de Collection tienen que ser de uno de los otros tipos compatibles o se tienen que poder serializar en un Blob.
  • Sin matrices. Como ya se ha mencionado antes, el almacén de datos admitirá valores de tipo Collection, así que utiliza este tipo en vez de matrices.
  • El almacén de datos no distingue los tamaños de los números, como podrían ser int32 e int64. Por ejemplo, si conservas una lista de enteros, se almacena como List<Long>.

Por supuesto, no estás obligado a utilizar el almacén de datos. Una de las ventajas de App Engine es que puedes mezclar y combinar muchos servicios en la nube. Por ejemplo, tienes JPA de Cloud SQL (que sería una muy buena opción) disponible para usarla. Como alternativa, puedes explorar Google BigQuery, si es el tipo de almacenamiento de datos que necesitas. Por último, Google Cloud Storage también es una buena manera de almacenar una gran variedad de tipos de datos.

Mensajería en la nube de Google

Las finalidades básicas de la mensajería en la nube de Google son bastante sencillas de utilizar. Todos los problemas conocidos están documentados en esta página.

Para los desarrolladores de Android primerizos

Si eres nuevo en Android, verás que Cloud Endpoints hace que todo sea mucho más sencillo y, además, te permite centrar tus energías en la propia aplicación, por ejemplo:

  • Haz llamadas a tu API de manera asíncrona en subprocesos que no sean de la interfaz de usuario. Android tiene en ejecución un subproceso principal (a menudo llamado "subproceso de la interfaz de usuario"), por lo que no es conveniente que se estanque y quede a la espera de respuestas.
    • Las tareas asíncronas, AsyncTasks, son muy útiles. Ten en cuenta que las AsyncTasks que están vinculadas a una actividad pueden quedar destruidas cuando se destruye la actividad, así que depende de ti garantizar que este tipo de comportamiento sea intencionado. Busca artículos relacionados, por ejemplo, con la rotación de la pantalla de AsyncTasks y es probable que encuentres muchas ideas para hacer frente a esta situación.
    • Si tienes que utilizar un objeto que la tarea tiene que ofrecer como resultado, utiliza el método de anulación OnPostExecute de la tarea. Si solicitas un objeto de Lobster a tu API mediante una AsyncTask y, justo después, intentas usarlo en tu próxima línea de código (por ejemplo, Lobster lobster = new GetLobsterAsyncTask().execute(lobsterName); lobster.dance();), recibirás una excepción. Es decir, no solo no funcionará, sino que lanzará una excepción NullPointerException.
    • Si quieres que la interfaz de usuario espere hasta que vuelva la tarea, usa ProgressDialog en los métodos OnPreExecute y OnPostExecute.
  • Haz un seguimiento de la actividad y de los ciclos de vida del fragmento. Si tienes problemas con esto al principio, anula un número de métodos de ciclo de vida importantes y coloca declaraciones de registro de eliminación de bugs en ellos; solo para ver si tus actividades se ponen en pausa, se reactivan, se destruyen, etc., y cuándo. En esta página tienes una lista completa de estos métodos, si los necesitas.
    • Por ejemplo, cuando inicias una segunda actividad con startActivityForResult, podrás ver que el primer obstáculo con el que se encuentra el ciclo de vida, cuando esa segunda actividad termina, es onActivityResult en la primera actividad, incluso antes de onCreate/Resume.
  • Aunque se inicie una nueva actividad en medio de un método, no se evita que se ejecute el resto de la línea de lógica. Esto incumbe a startActivity, startActivityForResult o a cualquier otra cosa. Aunque no lo puedas ver, se completa el método y cualquier otro método al que llame. La actividad que contiene el método en ejecución no desaparece hasta que la destruyas o hasta que la extraigas de la pila.
  • A veces, solo tienes que eliminar el archivo R.java. O, dicho de otro modo, limpia el proyecto y vuelve a crearlo. De vez en cuando te encontrarás con algunos problemas que no te permitan enviar Lobster a Tree. Puede que no te parezca factible, pero quizá ese problema quiera decir que el archivo R.java, que contiene punteros que señalan a los valores definidos en tus recursos o en tu carpeta, se ha confundido. Si vuelves a crearlo, en teoría el problema se solucionará.
  • Las reglas de diseños de Android no son reglas de CSS. A menudo pueden parecerse mucho, por lo que te resultarán familiares. Sin embargo, a veces hay pequeñas diferencias. Por ejemplo, los márgenes no se minimizan en Android, así que revisa las características antes de invertir mucho esfuerzo en el diseño.
  • Dalvik en realidad no es Java. Sin embargo, Dalvik sí utiliza sintaxis de Java, y Android es compatible con un gran subconjunto de bibliotecas estándar de Java, como por ejemplo java.util y java.io. No puedes importar cualquier biblioteca de Java de terceros y esperar que funcione bien con Android.
  • Si creas una nueva actividad con el asistente de creación de Eclipse, se generan o modifican cinco archivos:
    • AndroidManifest.xml (añade una nueva entrada <activity>).
    • res/values/strings.xml (añade una cadena de título para la actividad).
    • res/menu/your_activity.xml (crea este archivo).
    • res/layout/your_activity.xml (crea este archivo).
    • YourActivity.java (crea este archivo con algunos métodos de inicio, como la anulación de onCreate).
  • Utiliza la clase Logger estándar para iniciar sesión en App Engine. El resultado se puede ver en la consola de App Engine. Android utiliza su propio inicio de sesión y su resultado se puede ver en una de las vistas de Eclipse llamada "LogCat".
  • Si te encuentras con errores EOFException ocasionales, es muy probable que se deban a las conexiones keepAlive en Android. En StackOverflow puedes encontrar varias soluciones.

Apunte final

Has leído muchos de los detalles que es posible que te encuentres, pero la mejor manera de aprender es tirarte de cabeza. El servicio de Cloud Endpoints es sumamente flexible e intuitivo. Asume la carga que supone desarrollar protocolos e interacciones cliente-servidor, y te permite centrarte en el desarrollo de la lógica de tu aplicación. Lo mejor es que no molesta; cuando ya hayas empezado a desarrollar tu aplicación, verás que conseguir que tu cliente se comunique con tu servidor es tan fácil que te olvidarás de que no siempre fue tan sencillo.

¡Buena suerte!

Uso de @ApiMethod

Hay una tabla maravillosa en esta página que te guiará en la anotación de tus métodos de la API de Java. A continuación tienes algunos detalles adicionales:

La ruta del URI REST de tu extremo prefiere utilizar estos atributos y en el siguiente orden:

  1. Tu atributo path
    • En path = lobsters: "lobsters" se convertirá en tu ruta de REST.
  2. La segunda parte de tu atributo "name"
    • En name = lobsters.assignName: "assignName" se convertirá en tu ruta de REST.
  3. El nombre de tu método
    • En public void assignName(Lobster lobster): "assignName" se convertirá en tu ruta de REST.

En cuanto a tu atributo name, tienes que considerar que la primera parte de este atributo es tu clase de recurso. La segunda, por su parte, es el método que quieres realizar. A continuación hay un par de ejemplos del atributo name:

  • name = lobsters.assignName en tu backend se usará con service.lobsters().assignName(param) en tu código de cliente.
  • name = lobsters en tu backend se usará con service.lobsters(param) en tu código de cliente.

Ejemplo 1

Con esas reglas en mente, aquí tienes un ejemplo habitual del aspecto que puede tener un método en tu backend de App Engine. Ten en cuenta que el recurso, lobster, se ha pluralizado:

@ApiMethod(path = “lobsters/assign/[a]name}[a]”, name = "lobsters.assignName",
httpMethod = HttpMethod.POST)
public void makeLobsterANewName(String name) {...}

A continuación, la biblioteca cliente generará el siguiente código para utilizarlo en el cliente:

public Lobsters lobsters() {...}
  public class Lobsters {
  public AssignName assignName(String name) {...}
  public class AssignName extends CloudmazeRequest {
    private static final String REST_PATH = “lobsters/assign/{name}”
    protected AssignName(String name) {...}
  }
}

Y, para utilizar el extremo, puedes escribir el siguiente código:

String name = “Bob the Lobster”;
service.lobsters().assignName(name).execute();

Ejemplo 2

Aquí tienes un ejemplo habitual de un método tipo CRUD más sencillo para un recurso de Lobster en tu backend de App Engine. Ten en cuenta que el recurso, lobster, se ha pluralizado y que no está el atributo HttpMethod porque el método "update" utiliza HttpMethod.PUT de manera predeterminada:

@ApiMethod(path = “lobsters”, name = "lobsters.update")
public void updateLobster(Lobster lobster) {...}

A continuación, la biblioteca cliente genera el siguiente código para utilizarlo en el cliente:

public Lobsters lobsters() {...}
  public class Lobsters {
  public Update update(String name) {...}
  public class Update extends CloudmazeRequest {
    private static final String REST_PATH = “lobsters”
    protected Update(String name) {...}
  }
}

Para utilizar el extremo, puedes escribir el siguiente código:

Lobster bobLobster = new Lobster();
service.lobsters().update(bobLobster).execute();
